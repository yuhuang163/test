# Qproduct（产品串口 · 仪器侧协议）使用说明

帧定义与测试步骤以仓库根目录 **`docs/测试.md`** 为准；本类只做 **组帧、写串口、对已有接收数据做解析**，**不负责**从串口里替你「一直读」——读缓冲由工站 / `test_base` 等上层汇总后再调用下面的静态解析接口。

---

## 1. 对象从哪来

继承 **`test_base`** 的工站里已有：

- **`QSerialPort* productSerialPort`**：硬件串口（需先 **`openProductSerialPort()`** 成功）；
- **`Qproduct* product`**：绑定同一 `productSerialPort`，用于 **`writeRaw` / `writeHex`**。

`factory_analyzer` 里同样有 **`productSerialPort` + `product`**，用法相同。

---

## 2. 发送（写串口）

串口必须已 **`open`**。失败时建议把 **`QString err`** 打日志。

| 方式 | 说明 |
|------|------|
| **`product->writeRaw(frame, &err)`** | 写入一帧 **`QByteArray`** |
| **`product->writeHex("01030C00", &err)`** | 无空格十六进制字符串转字节再写（非法 hex 会失败） |

常用固定命令（与 **`docs/测试.md`** 一致，也可用 **`writeHex`** 写同内容）：

| API | 含义 |
|-----|------|
| **`Qproduct::cmdReset()`** | 复位命令（发） |
| **`Qproduct::buildStartReceiveCmd2402Ble1M()`** 等 6 个 | 各频段 / PHY 的「开始接收」命令（发） |
| **`Qproduct::cmdStopReceive()`** | 停止接收命令（发） |

六个 **`buildStartReceiveCmd*`** 对应文档中的 2402/2440/2480 × BLE 1M / 2M，按当前测试项选用其一即可。

典型顺序：**复位 → 开始接收 →（业务上等待仪器发完包）→ 停止接收**。

---

## 3. 接收与应答判断（不发送，只比对）

下列为 **期望从串口读到的应答模板**，用于在 **`QByteArray rx`**（你累积的接收数据）里查找，**不要**当发送数据写出去。

| API | 含义 |
|-----|------|
| **`Qproduct::ackReset()`** | 复位后的应答 |
| **`Qproduct::ackStartReceive()`** | 「开始接收」后的应答 |
| **`Qproduct::prefixStopReceive()`** | 「停止接收」应答的固定前缀，后面跟 **2 字节小端**收包数 |

判断是否收到某段应答（允许前面有杂字节 / 粘包，只要 **`rx` 里出现过整段**即真）：

```cpp
if (Qproduct::responseContainsPrefix(rx, Qproduct::ackReset())) {
    // 已看到复位应答
}
if (Qproduct::responseContainsPrefix(rx, Qproduct::ackStartReceive())) {
    // 已看到开始接收应答
}
```

---

## 4. 停止接收后：收包数与 PER

在包含停止应答的 **`rx`** 上：

```cpp
const int recvPkts = Qproduct::parseStopReceivePacketCountLe(rx);
if (recvPkts < 0) {
    // 未找到前缀或长度不够，尚未形成完整停止应答
} else {
    const int instrumentSend = /* 仪器本包次发包总数，来自配置/仪器/业务，勿在公共库里写死 */;
    const double per = Qproduct::computePer(instrumentSend, recvPkts);
}
```

**`computePer`**：\((\text{仪器发包数} - \text{解析出的收包数}) / \text{仪器发包数}\)。

---

## 5. 与 `test_base` 里缓冲的关系（重要）

默认路径：**`productSerialPort` 的 `readyRead` → `productSerialPortBuf` → 定时触发 `readProductSerialPortData` → 末尾会 `productSerialPortBuf.clear()`**。

**`test_base::readProductSerialPortData`（及 `factory_analyzer` 同名函数）在清空 `productSerialPortBuf` 之前，已调用 `product->parseCmd(dataTemp)`**（与 **`jig->parseCmd(dataTemp)`** 同形），把本批聚合数据追加进 **`Qproduct` 内累积缓冲**；仪器侧数据量不大，为直接 **`append`**。

工站里可直接用：

- **`product->productSerialRxAccum()`** 作为 `responseContainsPrefix` / `parseStopReceivePacketCountLe` 的入参；
- 新一轮仪器流程开始前调用 **`product->clearProductSerialRxAccum()`**（**关闭 / 重开产品串口**时公共代码也会清空，避免旧口数据串台）。

若仍要在工站里 **再** 维护一份自己的 `QByteArray`，可在 **`processReceivedData(data)`** 里继续 `append(data)`，与 `Qproduct` 累积并行；一般 **只读 `productSerialRxAccum()` 即可**。

**不建议**在业务里再对同一 `productSerialPort` 做 **`waitForReadyRead` + `readAll()`** 与现有 `readyRead` 槽 **抢读**，易半包或读空。

---

## 6. 其它静态工具

| API | 说明 |
|-----|------|
| **`Qproduct::hexToBytes(QString)`** | 无空格 hex → `QByteArray`，非法返回空 |
| **`Qproduct::bytesToHex(QByteArray)`** | 二进制 → 小写 hex 字符串（便于日志） |

## 7. 最小代码片段（拼接思路）

```cpp
QString err;
if (!product->writeRaw(Qproduct::cmdReset(), &err)) {
    showlog(QStringLiteral("产品串口写失败：%1").arg(err));
    return;
}
// 发复位后，在步骤/定时器里判断（数据已由 readProductSerialPortData 写入 product 累积）：
if (Qproduct::responseContainsPrefix(product->productSerialRxAccum(), Qproduct::ackReset())) {
    // …
}

if (!product->writeRaw(Qproduct::buildStartReceiveCmd2402Ble1M(), &err)) { /* ... */ }
// 判断 ackStartReceive：Qproduct::responseContainsPrefix(product->productSerialRxAccum(), Qproduct::ackStartReceive())

// 等待仪器发完包后：
if (!product->writeRaw(Qproduct::cmdStopReceive(), &err)) { /* ... */ }
// 发停止接收后：
const int n = Qproduct::parseStopReceivePacketCountLe(product->productSerialRxAccum());
```

具体等待方式、超时、`instrumentSend` 来源以各工站流程为准。
