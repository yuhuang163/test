# new_product_test — 自动生成提交说明（GitHub Copilot）

根据当前 **git diff / 暂存变更**，生成提交信息时请严格遵守下列格式（与 `my_set/AbIni.h` 中 `[XXX_VER]` 注释风格一致）。

## 输出必须遵循的模板（先选型再填空）

**流程**：先看暂存路径属于哪一类 → **从下表选一个最贴近的模板** → **只替换 `<…>` 里的中文说明**，**不得**改掉 **`[XXX_VER]`** 宏名、**不得**把整段改成英文长句、**不得**自拟与模板差异过大的结构。

| 情形 | 套用的模板 |
|------|------------|
| 只改**一个** `work_station/<子目录>/`（无 qtuple / 无 mainwindow / 无 docs 业务混改） | **模板 A** |
| 改**多个**工站目录 | **模板 B**（同一行）或 **模板 C**（分行） |
| **工站** + **`docs/` 等脚本** 同提交 | **模板 D** |
| **工站** + **`agreement/qtuple/`**（须单独写 FREE） | **模板 E** |
| **仅** `docs/`、`.githooks`、`validate_commit_message.py`、`.copilot*` 等，**无** `work_station/` 业务 | **模板 F** |
| **`mainwindow` / 根目录产测主流程** | **模板 G** |
| 需要 **feat/fix/docs/chore** 前缀 | **模板 H** 与上列之一组合 |

**模板 A（单工站，一行）**  
`[与路径对应的XXX_VER] <工站简称>：<用中文逗号写清与 diff 一致的改动，勿写未改动模块>`  
例：`[CAMERA_VER] 摄像测试工站：更新测试结果 CSV 保存目录，调整结果区背景色`

**模板 B（多工站，同一行，`；` 分隔块）**  
`[VER1] <说明1>；[VER2] <说明2>；[VER3] <说明3>`  
（每个 `VER` 须与本次路径、AbIni 一致）

**模板 C（多工站，分行）**  
第一行：`[VER1] <说明1>`  
空一行，第二行：`[VER2] <说明2>`（按需继续）

**模板 D（工站 + docs/脚本）**  
`[工站对应_VER] <工站改动>，<脚本路径或文件名>：<脚本改动>`  
例：`[AGE_VER] 老化界面修正背景色，docs/生成版本说明.py：统一列表序号格式`

**模板 E（工站 + 三元组 / qtuple）**  
`[工站_VER] <仅工站目录下的改动>；[FREE_VER] 三元组服务：<仅 qtuple 相关改动>`  
（**禁止**把三元组写进工站句却不出现 `[FREE_VER]`）

**模板 F（纯工具/文档/元信息，无任何 `[XXX_VER]`）**  
`<2～4 句简体中文，可带文件路径，说明改了什么>`  
例：`docs/validate_commit_message.py：允许同一行多标签；生成版本说明.py：解析逻辑与分号分隔一致`  
**禁止**写 `[FREE_VER]`「三元组」等。

**模板 G（主窗口 / 产测主流程）**  
`[DEBUG_VER] <与 mainwindow/根目录主流程相关的具体改动>`

**模板 H（可选标题行，与 A～G 组合）**  
第一行：`feat: <概括>` 或 `fix:` / `docs:` / `chore:`（见 `docs/Git提交类型前缀说明.md`）  
空一行，下面接 **模板 A～G** 的正文（含 `[XXX_VER]` 的段落）。**不要**在英文 `feat:` 那一行里塞 `[XXX_VER]`。

---

## 语言（必须简体中文）

- **除** **`[XXX_VER]` 宏名**（与 `AbIni.h` 一致、英文大写）及少量无法替代的技术词（如 **CSV**、**UI**、具体文件名）外，提交说明**主体须为简体中文**。
- **禁止**输出整句、整段英文说明。Copilot 易默认生成英文（如 `Update UI styles and file paths: ...`），在本仓库**一律视为错误**；须改写为中文句式。
- **错误示例**：`Update UI styles and file paths: changed test result background color and updated CSV folder path for camera test.`（无 `[XXX_VER]` 且全英文）
- **正确示例**（diff 含 `work_station/camera/`）：`[CAMERA_VER] 摄像测试工站：调整测试结果背景色，更新 CSV 输出目录路径`

## 多条 `[XXX_VER]` 的排版（允许同一行）

- **允许**在同一条提交说明里出现多个 `[XXX_VER]`，且**可以写在同一行**：块与块之间用 **`；`**（推荐）或 **`;`** 分隔，例如：  
  `[AGE_VER] 老化工站：…；[CAMERA_VER] 摄像测试工站：…；[KEY_VER] 按键测试：…`
- **亦允许**每个标签**独占一行**（仅换行、不用分号接下一标签），可读性更好时优先。
- **仍禁止**：把不同模块的要点**混写进同一个** `[XXX_VER]` 的说明里却**不单独写**应有的另一宏（例如改了 `agreement/qtuple` 却只在 `[AGE_VER]` 文字里提三元组、**不出现** `[FREE_VER]`），见下「必须与 diff 一致」第 5 条。
- **三元组 / `agreement/qtuple` → `[FREE_VER]`**；**`[DEBUG_VER]` 仅用于 `mainwindow`/根目录产测主流程**。

## 必须与 diff 一致（禁止乱写模块、禁止错标签）

1. **只能写本次 diff 里真实改到的路径对应的内容**：若变更列表里**没有** `agreement/qtuple/`、`mainwindow.ui` 等，提交说明里**禁止**出现「三元组服务」「操作码映射」「上报格式」「主窗口/UI」等**未发生改动的描述**（模型易胡编，一律视为错误）。
2. **路径里出现了哪个工站目录，就必须出现对应 `[XXX_VER]`**：例如路径含 `work_station/ageing/` → **必须**写 **`[AGE_VER]`**；**禁止**在仅有老化工站（及同目录 `.ui`）改动时，用 **`[DEBUG_VER]`** 一笔带过或完全省略老化标签。
3. **错误示例**（实际只改了老化工站）：`[DEBUG_VER] 更新三元组服务，新增操作码中文映射，优化上报格式，调整UI样式` — **三重错误**：误用 `DEBUG_VER`、缺 `[AGE_VER]`、描述了 diff 中不存在的三元组/UI 改动。
4. **正确示例**（仅 `work_station/ageing/*`）：`[AGE_VER] 老化测试工站：……（与 ageing.cpp / ageing.ui 中真实修改一致的具体说明）`
5. **工站 + 公共模块 混合提交**（diff 里**既有** `work_station/<子目录>/` **又有** `agreement/`、`test_base`、`box_base` 等）：**禁止**只写一个 `[AGE_VER]`（或单一工站宏），却在**其说明文字里**夹写三元组/协议等公共改动，而**不单独出现**对应的 **`[FREE_VER]`**（或其它应有宏）。  
   - **第一块**只写该工站目录下真实改动；**第二块**须单独有 **`[FREE_VER] …`** 写三元组/qtuple 等（可与上一块**同一物理行**，用 **`；`** 分隔，或分两行）。  
   **错误示例**：仅 `[AGE_VER] 老化UI…，并新增三元组操作码映射…` — 全文无 `[FREE_VER]`，公共改动无法进自由工站行。  
   **正确示例**（同一行）：`[AGE_VER] 老化测试工站：优化界面与测试结果背景色；[FREE_VER] 三元组服务：操作码中文映射与显示（qtupleservice）`  
   **正确示例**（两行）：  
   `[AGE_VER] 老化测试工站：优化界面与测试结果背景色，增强可读性`  
   `[FREE_VER] 三元组服务：操作码中文映射与显示（qtupleservice）`

## 工站代码与 `docs/` / 工具脚本同提交（易漏工站标签）

模型常见错误：看到 **`docs/生成版本说明.py`** 就把整段提交当成「仅文档」，**漏写** `work_station/ageing/` 对应的 **`[AGE_VER]`**。

- **规则**：只要 diff 里**有** `work_station/<子目录>/` 下的代码或 `.ui` 变更，**无论是否同时**改了 `docs/`、`.md`、`.copilot-*`，都**必须**出现该工站对应 **`[XXX_VER]`**（如老化 → **`[AGE_VER]`**）。
- **推荐写法**（老化 + 脚本）：`[AGE_VER] 老化测试界面：修正背景色以提升可读性；docs/生成版本说明.py：统一序号格式以改善 Markdown 预览`（同一行用中文逗号串联两处真实改动即可）。
- **错误写法**：`优化版本说明生成脚本…；调整老化测试界面样式…` — **全文无 `[AGE_VER]`**。

## 仅改工具脚本 / 校验 / 钩子（易误套 `[FREE_VER]`）

模型常见错误：改了 **`docs/生成版本说明.py`、`docs/validate_commit_message.py`、`.githooks/`** 等，却在说明里写 **`[FREE_VER] 三元组服务：…`**（把「提交格式、同一行多标签、校验逻辑」当成三元组业务）。

- **规则**：若 diff **只有** `docs/`、`.githooks/`、`.copilot*`、`.cursor/`、`.github/` 等**文档与仓库元信息/工具脚本**（**没有** `agreement/qtuple/`、`work_station/` 下业务代码或 `.ui`），提交说明须 **2～4 句简体中文、不要任何 `[XXX_VER]`**。
- **禁止**用 **`[FREE_VER]`** 或文内 **「三元组服务」** 描述纯脚本/钩子/生成说明类改动（会错误写入 `上位机版本发布.md` 自由工站行）。
- **正确示例**：`docs/生成版本说明.py：优化多标签解析；validate_commit_message.py：放宽同一行多标签校验`（无方括号宏）
- **错误示例**：`[FREE_VER] 三元组服务：允许同一行分号…；docs/生成版本说明.py：…`（本次无 qtuple/自由工站业务代码）

## 判定顺序（必遵守，避免单工站改动生成错标签）

1. 先根据变更文件路径（等价于 `git diff --name-only` / 暂存文件列表）归类。
2. **先看是否含工站目录**：若出现 `work_station/<子目录>/` 下文件变更，**必须先**写对应该 `[XXX_VER]`；**禁止**因同时存在 `docs/`、工具脚本路径就整条说明走「无标签」。
3. **仅改单一工站**：若本次变更**全部**落在**同一个** `work_station/<子目录>/` 下（例如只有 `work_station/ageing/ageing.cpp`），且**没有**同时改动 `agreement/`、`work_station/test_base.cpp`、`work_station/box_base.cpp`、`mainwindow.cpp`、`mainwindow.ui`、根目录产测主流程等，则提交信息 **只写一行**对应宏，例如 **`[AGE_VER] …`**。**禁止**在这种情况下写 `[DEBUG_VER]`，**禁止**套用「对 `AbIni.h` 里每一个 `XXX_VER` 各写一行」。
4. **多工站或跨模块**：若同时改了多个 `work_station/*/` 子目录，或改了公共模块，再按下面表格与「公共模块」条款处理（公共模块才可能需要多行或全量 `XXX_VER`）。**若同一提交既改某一工站又改 `agreement/qtuple` 等公共路径**，须按上文「工站 + 公共模块 混合提交」**分别写出**对应 `[XXX_VER]` 块（可两行，或**同一行**用 **`；`** 分隔），**禁止**把公共要点只写在单一工站宏的说明文字里却不出现 `[FREE_VER]` 等应有块。
5. **仅文档/脚本/元信息**：仅当 **diff 中没有任何** `work_station/` 下业务代码或 `.ui`、也没有需按工站/协议进发布说明的实现变更时，才适用下表第一行（**不要**写 `[XXX_VER]`）。

## `agreement/qtuple` 与三元组：**禁止 `[AGREEMENT_VER]`**

- 本仓库 **`my_set/AbIni.h` 中没有 `#define AGREEMENT_VER`**，**禁止**在提交说明里写 **`[AGREEMENT_VER]`**（否则 `docs/生成版本说明.py` 无法归类）。模型容易把「协议/agreement」联想成该宏，**一律视为错误**。
- 变更仅在 **`agreement/qtuple/`**（如三元组服务、上报 body、操作码中文映射、检验项字符串）且**不采用**下文「公共模块 → 每个 `XXX_VER` 一行」时：请用 **`[FREE_VER] …` 一行**概括（三元组能力挂在**自由测试工站**；**禁止**用 `[DEBUG_VER]` 写纯三元组改动）。仅文档类可无标签。
- **错误示例**：`[AGREEMENT_VER] 新增三元组操作码中文映射…`；**错误**：`[DEBUG_VER] 三元组服务：…`（三元组应使用 **`[FREE_VER]`**）  
- **正确示例**：`[FREE_VER] 三元组服务：操作码中文映射、上报检验项构造与显示名查询（qtupleservice）`
## 先判断类型（避免误用 `[DEBUG_VER]`）

| 变更内容 | 提交信息怎么写 |
|----------|----------------|
| **仅文档**（`docs/*.md` 等）、**仅发布/工具脚本**（如 **`docs/生成版本说明.py`、`docs/validate_commit_message.py`**）、**仅钩子**（`.githooks/`）、**仅仓库元信息**（`.copilot-*`、`.cursor/` 等），**不改动**各工站运行时 C++/UI 业务逻辑 | 用 **2～4 句简体中文**直接说明改了什么即可，**不要**写 `[XXX_VER]`（**尤其不要**用 `[FREE_VER] 三元组服务` 概括脚本/校验改动，见「仅改工具脚本 / 校验 / 钩子」）。不要用 `[DEBUG_VER]` 充当「万能标签」。**若同提交还改了 `work_station/` 下代码或 `.ui`，不适用本行**，须写对应 `[XXX_VER]`（见「工站代码与 `docs/` / 工具脚本同提交」与「判定顺序」第 2、5 条）。 |
| **`mainwindow` 或根目录产测主程序/主流程**（真实影响「产品产测工具」行为） | **`[DEBUG_VER] …`** 一行或多行要点。 |
| **公共模块业务代码**（`work_station/test_base`、`box_base`、`agreement/…` 等，各工站共享） | 对 **`AbIni.h` 中每一个 `#define XXX_VER` 各写一行**（见下「例外」）。 |
| **仅某一工站目录**（变更全部在该工站子目录内，且无公共模块/主窗口混改） | **只写一行**对应 `[XXX_VER]`（见「路径 → 宏标签」），例如只改老化 → 仅 `[AGE_VER] …`。 |

说明：`docs/生成版本说明.py` 只会把带 **`[XXX_VER]`** 的句子写进 `docs/上位机版本发布.md` 的工站行；**纯文字提交不会出现在该表各工站括号里**，这是预期，不是漏生成。

## 格式

- **多条 `[XXX_VER]` 时**：遵守文首 **「多条 `[XXX_VER]` 的排版」**；可同一行用 **`；`** / **`;`** 分隔，或每行一个标签。
- **可多行**。需要进入「上位机版本发布」工站行的内容，每一行对应**工站或规则如下**，形式为：
  - `[XXX_VER] 改动要点1，改动要点2，改动要点3`
- **`XXX_VER`** 必须与 `my_set/AbIni.h` 里 **`#define XXX_VER`** 的宏名**完全一致**（例如 `PCBA_VER`、`QC_VER`、`PRESSURE_VER`、`AGE_VER`、`SINGLE_VER`、`DEBUG_VER`、`FREE_VER`、`IMU_VER` 等）。**禁止臆造** `AbIni.h` 中不存在的宏名（例如不要自行发明 `AGREEMENT_VER`、`PROTOCOL_VER`、`UNKNOWN_VER` 等）。
- **只写本次变更涉及的工站**；未改动的工站不要写。
- **例外（公共模块）**：若改动落在公共模块（`work_station/test_base`、`box_base`、`agreement/qset`、`agreement/qProtocol`、`agreement/qtuple` 等，凡各工站共享、会一并影响发布说明的代码），**且**本次变更路径**并非**「全部落在单一 `work_station/<子目录>/`」时，须对 **`my_set/AbIni.h` 中每一个已有 `#define XXX_VER` 宏各写一行**，每行格式仍为 `[XXX_VER] …`，且**每一行都要写上本次公共改动的说明**（各工站行文字可相同，把同一套要点逐行复用即可）。**推荐写在 Git 提交正文**：第一行可写简短总述，正文里按工站逐行列出 `[PCBA_VER] …`、`[QC_VER] …` 等（行数与 `AbIni.h` 中宏数量一致），`docs/生成版本说明.py` 会扫描**标题+正文全文**，使 `docs/上位机版本发布.md` 里**每个工站条目**都带上该说明。若除公共模块外还有**仅某一工站**的改动，只在对应 `[XXX_VER]` 那一行里合并写上专属要点即可。
- 描述使用**简体中文**；同一行内多条用**中文逗号「，」**分隔。
- 避免空话（如「修改代码」「优化」「更新」且不说明改了什么）。
- **不要**为此发明 `AbIni.h` 里不存在的 `[XXX_VER]`。

## 输出要求

- **直接输出可粘贴进 Git 提交框的正文**，不要前缀解释、不要 Markdown 代码块包裹。
- **结构**：**优先**按上文 **「输出必须遵循的模板」** 选型套壳，再填 `<…>`；禁止不套壳就输出大段英文或散漫描述。
- **语言**：须符合 **「语言（必须简体中文）」**；**禁止**整段英文提交说明。
- **反例**：仅 `[AGE_VER] …` 一段文字里概括了三元组改动，却**不出现** `[FREE_VER]`（见「必须与 diff 一致」第 5 条）。
- **反例**：不要写 `[AGREEMENT_VER] …`（仓库无此宏）；三元组 / `qtupleservice` 用 **`[FREE_VER]`**，**不要**写 `[DEBUG_VER] 三元组…`。
- **反例**：**禁止**在 diff 含 `work_station/ageing/` 却用 **`[DEBUG_VER]` 描述三元组**或漏写 `[AGE_VER]`；说明文字必须与 diff 文件一致（见「必须与 diff 一致」）。
- **反例**：**禁止**在 diff 含 `work_station/ageing/` 与 `docs/生成版本说明.py` 时写成无 `[AGE_VER]` 的长句；须含 `[AGE_VER]`（见「工站代码与 `docs/` / 工具脚本同提交」）。
- **反例**：**禁止**在仅改 `docs/生成版本说明.py`、`docs/validate_commit_message.py`、`.githooks/` 等时写 **`[FREE_VER] 三元组服务：…`** 或**任意** `[XXX_VER]`（见「仅改工具脚本 / 校验 / 钩子」）。

## 稳定落地（不靠模型记忆）

在本机启用 **`docs/validate_commit_message.py` + `.githooks/commit-msg`** 后，`git commit` 会自动按暂存路径校验是否漏写 `[XXX_VER]`、是否出现非法宏名等。配置一次：`git config core.hooksPath .githooks`。说明见 **`docs/上位机版本信息生成说明.md`** 中的「提交信息校验」小节。
