# new_product_test — 自动生成提交说明（GitHub Copilot）

根据当前 **git diff / 暂存变更**，生成提交信息时请严格遵守下列格式（与 `my_set/AbIni.h` 中 `[XXX_VER]` 注释风格一致）。

## 标签之间必须换行

- 凡有 **两条及以上** `[XXX_VER]`：**每一条单独一行**，**行与行之间只用换行分隔**（可先标题一行再正文，但**不得**把两个标签挤在同一行）。
- **禁止**用 **`；`** 或 **`;`** 在同一行里连接两条标签。  
  **错误**：`[AGE_VER] …；[FREE_VER] …`（同一行）  
  **正确**：第一行 `[AGE_VER] …`，**换行**，第二行写与 diff 对应的宏；**三元组 / `agreement/qtuple` → `[FREE_VER]`**，**`[DEBUG_VER]` 仅用于 `mainwindow`/根目录产测主流程**，不要用来写三元组。

## 必须与 diff 一致（禁止乱写模块、禁止错标签）

1. **只能写本次 diff 里真实改到的路径对应的内容**：若变更列表里**没有** `agreement/qtuple/`、`mainwindow.ui` 等，提交说明里**禁止**出现「三元组服务」「操作码映射」「上报格式」「主窗口/UI」等**未发生改动的描述**（模型易胡编，一律视为错误）。
2. **路径里出现了哪个工站目录，就必须出现对应 `[XXX_VER]`**：例如路径含 `work_station/ageing/` → **必须**写 **`[AGE_VER]`**；**禁止**在仅有老化工站（及同目录 `.ui`）改动时，用 **`[DEBUG_VER]`** 一笔带过或完全省略老化标签。
3. **错误示例**（实际只改了老化工站）：`[DEBUG_VER] 更新三元组服务，新增操作码中文映射，优化上报格式，调整UI样式` — **三重错误**：误用 `DEBUG_VER`、缺 `[AGE_VER]`、描述了 diff 中不存在的三元组/UI 改动。
4. **正确示例**（仅 `work_station/ageing/*`）：`[AGE_VER] 老化测试工站：……（与 ageing.cpp / ageing.ui 中真实修改一致的具体说明）`
5. **工站 + 公共模块 混合提交**（diff 里**既有** `work_station/<子目录>/` **又有** `agreement/`、`test_base`、`box_base` 等）：**禁止**用**一行** `[XXX_VER]` 把「工站专属」和「公共（如三元组、协议）」混写在一起。**必须拆成多行**：  
   - **第一块**只写该工站：`[AGE_VER] …` 里**只写**老化目录下真实改动，**不要**写三元组/协议；  
   - **第二块**单独写三元组相关：`agreement/qtuple`、三元组上报/检验项等 → **`[FREE_VER] …` 一行**（归属**自由测试工站**）；若启用「公共模块 → 每个 `XXX_VER` 一行」则按该条款。  
   **错误示例**：`[AGE_VER] 老化UI…；新增三元组操作码中文映射…` — 公共内容塞进 `AGE_VER`，发布说明里其它工站看不到公共改动，也违背「公共模块应单独标注」的预期。  
   **正确示例**（**中间为换行**，非分号；共两行）：  
   `[AGE_VER] 老化测试工站：优化界面与测试结果背景色，增强可读性`  
   `[FREE_VER] 三元组服务：操作码中文映射与显示（qtupleservice）`

## 工站代码与 `docs/` / 工具脚本同提交（易漏工站标签）

模型常见错误：看到 **`docs/生成版本说明.py`** 就把整段提交当成「仅文档」，**漏写** `work_station/ageing/` 对应的 **`[AGE_VER]`**。

- **规则**：只要 diff 里**有** `work_station/<子目录>/` 下的代码或 `.ui` 变更，**无论是否同时**改了 `docs/`、`.md`、`.copilot-*`，都**必须**出现该工站对应 **`[XXX_VER]`**（如老化 → **`[AGE_VER]`**）。
- **推荐写法**（老化 + 脚本）：`[AGE_VER] 老化测试界面：修正背景色以提升可读性；docs/生成版本说明.py：统一序号格式以改善 Markdown 预览`（同一行用中文逗号串联两处真实改动即可）。
- **错误写法**：`优化版本说明生成脚本…；调整老化测试界面样式…` — **全文无 `[AGE_VER]`**。

## 判定顺序（必遵守，避免单工站改动生成错标签）

1. 先根据变更文件路径（等价于 `git diff --name-only` / 暂存文件列表）归类。
2. **先看是否含工站目录**：若出现 `work_station/<子目录>/` 下文件变更，**必须先**写对应该 `[XXX_VER]`；**禁止**因同时存在 `docs/`、工具脚本路径就整条说明走「无标签」。
3. **仅改单一工站**：若本次变更**全部**落在**同一个** `work_station/<子目录>/` 下（例如只有 `work_station/ageing/ageing.cpp`），且**没有**同时改动 `agreement/`、`work_station/test_base.cpp`、`work_station/box_base.cpp`、`mainwindow.cpp`、`mainwindow.ui`、根目录产测主流程等，则提交信息 **只写一行**对应宏，例如 **`[AGE_VER] …`**。**禁止**在这种情况下写 `[DEBUG_VER]`，**禁止**套用「对 `AbIni.h` 里每一个 `XXX_VER` 各写一行」。
4. **多工站或跨模块**：若同时改了多个 `work_station/*/` 子目录，或改了公共模块，再按下面表格与「公共模块」条款处理（公共模块才可能需要多行或全量 `XXX_VER`）。**若同一提交既改某一工站又改 `agreement/qtuple` 等公共路径**，须按上文「工站 + 公共模块 混合提交」**拆开写**，不得把公共内容只写在工站那一行里。
5. **仅文档/脚本/元信息**：仅当 **diff 中没有任何** `work_station/` 下业务代码或 `.ui`、也没有需按工站/协议进发布说明的实现变更时，才适用下表第一行（**不要**写 `[XXX_VER]`）。

## `agreement/qtuple` 与三元组：**禁止 `[AGREEMENT_VER]`**

- 本仓库 **`my_set/AbIni.h` 中没有 `#define AGREEMENT_VER`**，**禁止**在提交说明里写 **`[AGREEMENT_VER]`**（否则 `docs/生成版本说明.py` 无法归类）。模型容易把「协议/agreement」联想成该宏，**一律视为错误**。
- 变更仅在 **`agreement/qtuple/`**（如三元组服务、上报 body、操作码中文映射、检验项字符串）且**不采用**下文「公共模块 → 每个 `XXX_VER` 一行」时：请用 **`[FREE_VER] …` 一行**概括（三元组能力挂在**自由测试工站**；**禁止**用 `[DEBUG_VER]` 写纯三元组改动）。仅文档类可无标签。
- **错误示例**：`[AGREEMENT_VER] 新增三元组操作码中文映射…`；**错误**：`[DEBUG_VER] 三元组服务：…`（三元组应使用 **`[FREE_VER]`**）  
- **正确示例**：`[FREE_VER] 三元组服务：操作码中文映射、上报检验项构造与显示名查询（qtupleservice）`
## 先判断类型（避免误用 `[DEBUG_VER]`）

| 变更内容 | 提交信息怎么写 |
|----------|----------------|
| **仅文档**（`docs/*.md` 等）、**仅发布/工具脚本**（如 `docs/生成版本说明.py`）、**仅仓库元信息**（`.copilot-*`、`.cursor/` 等），**不改动**各工站运行时 C++/UI 业务逻辑 | 用 **2～4 句简体中文**直接说明改了什么即可，**不要**写 `[XXX_VER]`。不要用 `[DEBUG_VER]` 充当「万能标签」。**若同提交还改了 `work_station/` 下代码或 `.ui`，不适用本行**，须写对应 `[XXX_VER]`（见「工站代码与 `docs/` / 工具脚本同提交」与「判定顺序」第 2、5 条）。 |
| **`mainwindow` 或根目录产测主程序/主流程**（真实影响「产品产测工具」行为） | **`[DEBUG_VER] …`** 一行或多行要点。 |
| **公共模块业务代码**（`work_station/test_base`、`box_base`、`agreement/…` 等，各工站共享） | 对 **`AbIni.h` 中每一个 `#define XXX_VER` 各写一行**（见下「例外」）。 |
| **仅某一工站目录**（变更全部在该工站子目录内，且无公共模块/主窗口混改） | **只写一行**对应 `[XXX_VER]`（见「路径 → 宏标签」），例如只改老化 → 仅 `[AGE_VER] …`。 |

说明：`docs/生成版本说明.py` 只会把带 **`[XXX_VER]`** 的句子写进 `docs/上位机版本发布.md` 的工站行；**纯文字提交不会出现在该表各工站括号里**，这是预期，不是漏生成。

## 格式

- **多条 `[XXX_VER]` 时**：遵守文首 **「标签之间必须换行」**；每条 `[XXX_VER] …` 独占一行。
- **可多行**。需要进入「上位机版本发布」工站行的内容，每一行对应**工站或规则如下**，形式为：
  - `[XXX_VER] 改动要点1，改动要点2，改动要点3`
- **`XXX_VER`** 必须与 `my_set/AbIni.h` 里 **`#define XXX_VER`** 的宏名**完全一致**（例如 `PCBA_VER`、`QC_VER`、`PRESSURE_VER`、`AGE_VER`、`SINGLE_VER`、`DEBUG_VER`、`FREE_VER`、`IMU_VER` 等）。**禁止臆造** `AbIni.h` 中不存在的宏名（例如不要自行发明 `AGREEMENT_VER`、`PROTOCOL_VER`、`UNKNOWN_VER` 等）。
- **只写本次变更涉及的工站**；未改动的工站不要写。
- **例外（公共模块）**：若改动落在公共模块（`work_station/test_base`、`box_base`、`agreement/qset`、`agreement/qProtocol`、`agreement/qtuple` 等，凡各工站共享、会一并影响发布说明的代码），**且**本次变更路径**并非**「全部落在单一 `work_station/<子目录>/`」时，须对 **`my_set/AbIni.h` 中每一个已有 `#define XXX_VER` 宏各写一行**，每行格式仍为 `[XXX_VER] …`，且**每一行都要写上本次公共改动的说明**（各工站行文字可相同，把同一套要点逐行复用即可）。**推荐写在 Git 提交正文**：第一行可写简短总述，正文里按工站逐行列出 `[PCBA_VER] …`、`[QC_VER] …` 等（行数与 `AbIni.h` 中宏数量一致），`docs/生成版本说明.py` 会扫描**标题+正文全文**，使 `docs/上位机版本发布.md` 里**每个工站条目**都带上该说明。若除公共模块外还有**仅某一工站**的改动，只在对应 `[XXX_VER]` 那一行里合并写上专属要点即可。
- 描述使用**简体中文**；同一行内多条用**中文逗号「，」**分隔。
- 避免空话（如「修改代码」「优化」「更新」且不说明改了什么）。
- **不要**为此发明 `AbIni.h` 里不存在的 `[XXX_VER]`。

## 输出要求

- **直接输出可粘贴进 Git 提交框的正文**，不要前缀解释、不要 Markdown 代码块包裹。
- **反例**：`[AGE_VER] …；[FREE_VER] …` 写在**同一行** — 违反「标签之间必须换行」。
- **反例**：**禁止**把三元组/协议等改动与工站改动挤在**同一行** `[AGE_VER]` 里；混合提交须 **`[AGE_VER]` 与 `[FREE_VER]`（三元组）分行**，或公共模块全量多行（见「必须与 diff 一致」第 5 条）。
- **反例**：不要写 `[AGREEMENT_VER] …`（仓库无此宏）；三元组 / `qtupleservice` 用 **`[FREE_VER]`**，**不要**写 `[DEBUG_VER] 三元组…`。
- **反例**：**禁止**在 diff 含 `work_station/ageing/` 却用 **`[DEBUG_VER]` 描述三元组**或漏写 `[AGE_VER]`；说明文字必须与 diff 文件一致（见「必须与 diff 一致」）。
- **反例**：**禁止**在 diff **同时含** `work_station/ageing/` 与 `docs/生成版本说明.py` 时，写成无 `[AGE_VER]` 的长句（模型易误判为「仅文档」）；须含 `[AGE_VER]`（见「工站代码与 `docs/` / 工具脚本同提交」）。

## 稳定落地（不靠模型记忆）
