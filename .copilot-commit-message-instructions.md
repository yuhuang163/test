# new_product_test — 自动生成提交说明（GitHub Copilot）

根据当前 **git diff / 暂存变更**，生成提交信息时请严格遵守下列格式（与 `my_set/AbIni.h` 中 `[XXX_VER]` 注释风格一致）。

## 先判断类型（避免误用 `[DEBUG_VER]`）

| 变更内容 | 提交信息怎么写 |
|----------|----------------|
| **仅文档**（`docs/*.md` 等）、**仅发布/工具脚本**（如 `docs/生成版本说明.py`）、**仅仓库元信息**（`.copilot-*`、`.cursor/` 等），**不改动**各工站运行时 C++/UI 业务逻辑 | 用 **2～4 句简体中文**直接说明改了什么即可，**不要**写 `[XXX_VER]`。不要用 `[DEBUG_VER]` 充当「万能标签」。 |
| **`mainwindow` 或根目录产测主程序/主流程**（真实影响「产品产测工具」行为） | **`[DEBUG_VER] …`** 一行或多行要点。 |
| **公共模块业务代码**（`work_station/test_base`、`box_base`、`agreement/…` 等，各工站共享） | 对 **`AbIni.h` 中每一个 `#define XXX_VER` 各写一行**（见下「例外」）。 |
| **某一工站目录** | 只写对应 `[XXX_VER]`（见「路径 → 宏标签」）。 |

说明：`docs/生成版本说明.py` 只会把带 **`[XXX_VER]`** 的句子写进 `docs/上位机版本发布.md` 的工站行；**纯文字提交不会出现在该表各工站括号里**，这是预期，不是漏生成。

## 格式

- **可多行**。需要进入「上位机版本发布」工站行的内容，每一行对应**工站或规则如下**，形式为：
  - `[XXX_VER] 改动要点1，改动要点2，改动要点3`
- **`XXX_VER`** 必须与 `my_set/AbIni.h` 里 **`#define XXX_VER`** 的宏名**完全一致**（例如 `PCBA_VER`、`QC_VER`、`PRESSURE_VER`、`AGE_VER`、`SINGLE_VER`、`DEBUG_VER`、`FREE_VER`、`IMU_VER` 等）。**禁止臆造** `AbIni.h` 中不存在的宏名（例如不要自行发明 `AGREEMENT_VER`、`PROTOCOL_VER`、`UNKNOWN_VER` 等）。
- **只写本次变更涉及的工站**；未改动的工站不要写。
- **例外（公共模块）**：若改动落在公共模块（`work_station/test_base`、`box_base`、`agreement/qset`、`agreement/qProtocol`、`agreement/qtuple` 等，凡各工站共享、会一并影响发布说明的代码），须对 **`my_set/AbIni.h` 中每一个已有 `#define XXX_VER` 宏各写一行**，每行格式仍为 `[XXX_VER] …`，且**每一行都要写上本次公共改动的说明**（各工站行文字可相同，把同一套要点逐行复用即可）。**推荐写在 Git 提交正文**：第一行可写简短总述，正文里按工站逐行列出 `[PCBA_VER] …`、`[QC_VER] …` 等（行数与 `AbIni.h` 中宏数量一致），`docs/生成版本说明.py` 会扫描**标题+正文全文**，使 `docs/上位机版本发布.md` 里**每个工站条目**都带上该说明。若除公共模块外还有**仅某一工站**的改动，只在对应 `[XXX_VER]` 那一行里合并写上专属要点即可。
- 描述使用**简体中文**；同一行内多条用**中文逗号「，」**分隔。
- 避免空话（如「修改代码」「优化」「更新」且不说明改了什么）。
- **不要**为此发明 `AbIni.h` 里不存在的 `[XXX_VER]`。

## 路径 → 宏标签（按变更文件路径优先判断）

- `work_station/pcba/` → `PCBA_VER`
- `work_station/quiescent_current/` → `QC_VER`
- `work_station/pressure/` → `PRESSURE_VER`
- `work_station/ageing/` → `AGE_VER`
- `work_station/wifi_ble/` → `SINGLE_VER`
- `work_station/freework/` → `FREE_VER`
- `work_station/imu/` → `IMU_VER`
- `work_station/key/` → `KEY_VER`
- `work_station/screen/` → `SCREEN_VER`
- `work_station/camera/` → `CAMERA_VER`
- `work_station/motor/` → `MOTOR_VER`
- `work_station/suction/` → `SUCTION_VER`
- `mainwindow` 或根目录产测主流程（且不属于上述单一工站目录）→ `DEBUG_VER`
- 其余宏（如 `LIGHT_VER`、`CAMERA_VER` 等）以 `AbIni.h` 中 `#define` 为准。

## 输出要求

- **直接输出可粘贴进 Git 提交框的正文**，不要前缀解释、不要 Markdown 代码块包裹。
- **反例**：仅更新《上位机版本信息生成说明》、优化 `生成版本说明.py` 的 Git 解析时，不要写 `[DEBUG_VER] 更新文档…`；应写无标签的纯文字说明（见上表第一行）。
