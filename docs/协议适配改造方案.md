# 协议适配改造方案

## 1. 背景

当前项目里，`qpb` 不只是一个“协议编解码模块”，它实际上同时承担了下面几类职责：

- 串口数据解析与组包
- CRC、AES、包头包尾等链路细节
- protobuf 结构的编解码
- 业务命令发送接口
- 业务响应解析
- 工站流程依赖的状态位维护
- UI 和工站层直接消费的信号分发

这会带来一个明显问题：如果现在要把所有 `qpb` 换成新协议，改动不会只停留在协议层，而会一路扩散到 `work_station`、`mainlogic`、`mainwindow` 等业务代码。

如果以后还会继续支持第 3 种、第 4 种协议，那么现在最值得做的不是“把 `qpb` 全量替换成新协议实现”，而是先把“协议”和“业务”之间切开，建立稳定的适配边界。

## 2. 当前代码里的主要耦合点

结合当前项目结构，耦合主要集中在下面几类位置：

- `agreement/qpb/qpb.h`
  `Qpb` 暴露了大量业务接口，例如 `set_motor_cali()`、`get_base_info()`、`set_sn()`、`get_battery()`，这说明业务层已经把 `Qpb` 当成“设备服务”在用，而不只是协议类。

- `agreement/qpb/qpb.cpp`
  `Qpb` 内部同时处理收包状态机、protobuf 编解码、命令分发、业务状态位更新、信号发送，单类职责过重。

- `work_station/test_base.h`
  `Qpb* pb` 被放进了工站基类，导致所有工站天然依赖 `Qpb`。

- `work_station/test_base.cpp`
  串口收到的数据会直接执行 `pb->parseCmd(dataTemp)`，说明数据入口已经和 `Qpb` 绑定。

- `work_station/*`
  多个工站直接调用 `pb->xxx()`，例如电机、IMU、PCBA、压感、老化、屏幕等流程都直接依赖 `Qpb` 的具体接口。

- `work_station/*`、`mainlogic.cpp`、`mainwindow.cpp`
  直接使用 `FacGetDevBaseInfo`、`FacDevInfo`、`FacUploadPresSensor`、`FacImuCalibResult` 这类 protobuf 生成类型，说明业务层已经依赖了 `qpb` 的数据模型。

- `test_base::sendCommandWithRetry()`
  目前重试与应答基于 `sendGetBrushResponse(int)` 这样的通用信号，抽象层次太低，也不利于后续切协议。

## 3. 这次改造的核心目标

这次协议替换建议同时满足 3 个目标：

- 先让“业务流程”不再直接依赖 `Qpb`
- 再让“业务数据结构”不再直接依赖 protobuf 生成体
- 最后把“协议实现”收敛成可插拔模块

最终希望做到：

- 工站层只关心“我要做什么”，不关心底层是 `qpb` 还是新协议
- 协议层只关心“怎么收发和编解码”，不关心电机、压感、IMU 这些业务流程
- 新增协议时，主要新增一个适配器，而不是全项目改一遍

## 4. 推荐的分层方式

建议把当前结构拆成 4 层。

### 4.1 Transport 层

职责：

- 串口打开、关闭、读写
- 原始字节流收发
- 超时、重连、链路错误

建议接口示意：

```cpp
class ITransport : public QObject {
    Q_OBJECT
public:
    virtual ~ITransport() = default;
    virtual bool open() = 0;
    virtual void close() = 0;
    virtual qint64 write(const QByteArray& data) = 0;

signals:
    void dataReceived(const QByteArray& data);
    void errorOccurred(const QString& error);
};
```

这一层以后无论是串口、USB、TCP，还是别的链路，都可以单独适配。

### 4.2 Protocol Codec 层

职责：

- 原始字节流拆包
- 包头、长度、CRC、加解密
- 协议对象与字节流的转换

这一层只处理“协议格式”，不要放业务语义。

建议接口示意：

```cpp
struct ProtocolFrame {
    int command;
    QByteArray payload;
};

class IProtocolCodec {
public:
    virtual ~IProtocolCodec() = default;
    virtual QList<ProtocolFrame> decode(const QByteArray& bytes) = 0;
    virtual QByteArray encode(const ProtocolFrame& frame) = 0;
};
```

如果新协议和旧协议的包格式完全不一样，就分别做：

- `QpbProtocolCodec`
- `NewProtocolCodec`

### 4.3 Device Service 层

职责：

- 向上提供统一业务动作
- 向下调用具体协议适配器
- 屏蔽不同协议的命令字、字段差异

这一层是后续适配多协议的关键层，建议由它替代现在业务中直接使用的 `Qpb`。

建议接口示意：

```cpp
class IDeviceService : public QObject {
    Q_OBJECT
public:
    virtual ~IDeviceService() = default;

    virtual void getBaseInfo() = 0;
    virtual void getBattery() = 0;
    virtual void getSn(int type) = 0;
    virtual void setForbidSleep(bool enable) = 0;
    virtual void setMotorCali(int step) = 0;
    virtual void setImuCollect(bool enable) = 0;

signals:
    void baseInfoReady(const DeviceBaseInfo& data);
    void batteryReady(const BatteryInfo& data);
    void snReady(const DeviceSnInfo& data);
    void operationFinished(const DeviceOperationResult& result);
};
```

### 4.4 Domain / Workstation 层

职责：

- 工站状态机
- 界面刷新
- 测试流程编排
- MES 交互

这一层只依赖 `IDeviceService` 和项目自己的业务数据结构，不直接依赖 protobuf 生成代码。

## 5. 一定要补上的一层：项目自己的 DTO

目前业务层大量直接使用：

- `FacGetDevBaseInfo`
- `FacDevInfo`
- `FacUploadPresSensor`
- `FacImuCalibResult`
- `FacGetPeriphState`

这会导致一个问题：哪怕协议接口形式统一了，只要新协议不用这套 protobuf 结构，业务代码仍然要大改。

所以建议新增一组“项目内部统一数据结构”，也就是 DTO。

例如：

```cpp
struct DeviceBaseInfo {
    QString macAddress;
    QString firmwareVersion;
    QString hardwareVersion;
    QString productName;
};

struct BatteryInfo {
    int percent = 0;
    double voltage = 0.0;
    bool charging = false;
};

struct DeviceSnInfo {
    QString tailSn;
    QString subPid;
};
```

然后：

- `Qpb` 收到 protobuf 数据后，先转换成 DTO
- UI、工站、业务流程只使用 DTO
- 新协议只要也能转换成相同 DTO，就可以无缝复用上层逻辑

这是后续新增协议时最省成本的一步。

## 6. 推荐的适配模式

最适合当前项目的不是“到处 if/else 判断当前协议”，而是“接口 + 适配器 + 工厂”。

建议结构如下：

- `IDeviceService`
  统一业务接口

- `QpbDeviceService`
  基于现有 `qpb` 的适配实现

- `NewProtocolDeviceService`
  基于新协议的适配实现

- `ProtocolServiceFactory`
  根据配置、设备类型或编译选项生成具体实现

示意：

```cpp
enum class ProtocolType {
    Qpb,
    NewProtocol
};

class ProtocolServiceFactory {
public:
    static IDeviceService* create(ProtocolType type, QObject* parent = nullptr);
};
```

这样以后新增协议时，主要新增：

- 一个 codec
- 一个 device service adapter
- 少量工厂注册代码

而不是把所有工站流程再改一遍。

## 7. 对当前项目最现实的改法

不建议一次性重写全部协议链路。当前代码历史包袱比较重，最稳妥的方式是“分阶段替换”。

### 阶段 1：先包一层，不动大业务

目标：

- 保留 `Qpb` 现有实现
- 在外面新增 `IDeviceService` 和 `QpbDeviceService`
- 让工站先从直接依赖 `Qpb`，改成依赖 `IDeviceService`

这一步里，`QpbDeviceService` 内部可以先继续调用老的 `Qpb`：

```cpp
class QpbDeviceService : public IDeviceService {
    Q_OBJECT
public:
    explicit QpbDeviceService(Qpb* pb, QObject* parent = nullptr);

    void getBaseInfo() override;
    void getBattery() override;
    void getSn(int type) override;
    void setForbidSleep(bool enable) override;

private:
    Qpb* pb_;
};
```

好处：

- 业务改动范围可控
- 可以先把“协议依赖点”集中收口
- 即使新协议还没写完，也不会阻塞现有功能

### 阶段 2：把 protobuf 类型挡在适配层内

目标：

- `QpbDeviceService` 对外只发 DTO
- `work_station`、`mainlogic`、`mainwindow` 不再直接使用 `Fac*`

这是非常关键的一步。因为只做完第 1 阶段，业务虽然不直接调 `Qpb` 了，但依然会被 protobuf 结构绑住。

### 阶段 3：拆 `Qpb` 内部职责

把现在的 `Qpb` 再往下拆成：

- `QpbTransport` 或复用现有串口对象
- `QpbProtocolCodec`
- `QpbMessageMapper`
- `QpbDeviceService`

其中：

- `QpbProtocolCodec` 负责编解码
- `QpbMessageMapper` 负责把 protobuf 消息转成 DTO / 事件
- `QpbDeviceService` 负责暴露统一业务接口

### 阶段 4：接入新协议

此时新增：

- `NewProtocolCodec`
- `NewProtocolMessageMapper`
- `NewProtocolDeviceService`

如果上面几步做得彻底，那么多数工站逻辑可以不改，或者只改极少量与能力差异有关的地方。

## 8. 建议优先改哪些文件

建议优先从依赖最核心、影响最广的地方下手。

第一批建议改造文件：

- `work_station/test_base.h`
  把 `Qpb* pb` 的对外依赖，逐步替换成 `IDeviceService* deviceService`

- `work_station/test_base.cpp`
  把 `pb` 的信号连接迁移到统一业务信号

- `agreement/qpb/qpb.h`
  不建议继续加新业务接口，避免 `Qpb` 继续膨胀

- 新增 `agreement/protocol/` 或 `core/device/`
  放统一接口、DTO、工厂、适配器

- `work_station/motor/motor.cpp`
  适合作为第一个试点工站，因为它对 `pb` 的依赖非常典型

建议新增目录示例：

- `core/device/i_device_service.h`
- `core/device/device_dto.h`
- `core/device/device_events.h`
- `core/device/protocol_service_factory.h`
- `agreement/qpb/qpb_device_service.h`
- `agreement/qpb/qpb_device_service.cpp`
- `agreement/new_protocol/new_protocol_device_service.h`
- `agreement/new_protocol/new_protocol_device_service.cpp`

## 9. 关于重试和应答机制的建议

当前 `sendCommandWithRetry()` 依赖一个比较泛化的 `sendGetBrushResponse(int)`，这在单协议阶段还能勉强工作，但多协议后会越来越难维护。

建议后续改成“按操作类型回调结果”。

例如：

```cpp
enum class DeviceOperation {
    GetBaseInfo,
    GetBattery,
    SetForbidSleep,
    SetMotorCali,
    SetImuCollect
};

struct DeviceOperationResult {
    DeviceOperation operation;
    bool success;
    QString message;
};
```

然后让统一服务层发：

- `operationFinished(DeviceOperationResult result)`

这样 `sendCommandWithRetry()` 就不需要猜“当前这个响应是不是我要等的那个响应”，可维护性会高很多。

## 10. 为以后新增协议预留的设计点

为了让后续新增协议更方便，建议一开始就把下面几个点做进去：

- 协议能力声明
  不同协议未必支持完全相同的命令，建议加能力查询，例如 `supports(DeviceOperation::SetMotorCali)`。

- 统一错误模型
  不同协议的失败码最终都映射成项目内统一错误码或错误信息。

- 配置化选择协议
  可通过 ini、命令行参数或设备型号决定当前使用哪种协议。

- 避免上层出现协议分支
  尽量不要在工站里写 `if (protocol == qpb)` 这种逻辑，应该下沉到 adapter 内部。

- 一类业务动作只保留一个入口
  例如“获取基础信息”在上层永远只有 `getBaseInfo()`，不要让不同协议把差异暴露给业务层。

## 11. 我对这次替换的具体建议

如果你的目标是“先尽快把 `qpb` 换成新协议，同时又不想以后继续痛苦扩展”，我建议按照下面顺序推进：

1. 先定义统一接口 `IDeviceService`
2. 先定义统一 DTO，尤其是基础信息、电池、SN、外设状态、压感、IMU 这些高频数据
3. 先写 `QpbDeviceService`，把老 `Qpb` 包进去
4. 把 `test_base` 改成依赖 `IDeviceService`
5. 先挑一个工站试点迁移，优先建议 `motor`
6. 试点跑通后，再批量迁移其它工站
7. 最后再接入 `NewProtocolDeviceService`

这个顺序的好处是：

- 风险最小
- 可以边迁移边验证
- 老协议和新协议可以短期共存
- 后续再加协议时基本复用同一套上层代码

## 12. 不建议的做法

下面几种做法短期看似快，长期成本会更高：

- 直接把 `Qpb` 里的 protobuf 字段名替换成新协议字段名
- 在现有 `Qpb` 类里继续塞新协议分支
- 在工站代码里大量增加 `if/else` 判断当前协议
- 继续让 UI 和工站直接依赖 protobuf 生成类型
- 继续用“一个通用响应信号”承接所有命令回包

这些做法会让旧问题从 `qpb` 复制到“新协议版本的 qpb”里，后面还会再痛一次。

## 13. 最终建议结论

从当前项目现状看，最合适的方向不是“替换一个协议类”，而是：

- 把 `Qpb` 从“业务核心类”降级成“旧协议适配器”
- 在它上面建立统一设备服务接口
- 在它和业务之间建立项目自己的 DTO
- 让工站和 UI 只依赖统一接口，不依赖具体协议

这样做之后：

- 这次切新协议会更可控
- 以后新增别的协议也只需要补一个新的 adapter
- 业务流程和 UI 的复用率会明显提高
- 协议层的风险会被限制在较小范围内

## 14. 下一步可直接落地的最小任务

如果要开始真正改代码，建议第一轮只做下面 4 件事：

1. 新增 `IDeviceService`
2. 新增 `device_dto.h`
3. 新增 `QpbDeviceService`，内部先桥接老 `Qpb`
4. 把 `test_base` 从 `Qpb* pb` 改成 `IDeviceService* deviceService`

这 4 步做完，整个项目就算正式具备“多协议适配”的骨架了，后续再接新协议会轻松很多。
