# 提交代码与提交说明（工作流说明）

本文归纳当前仓库里与 **Git 提交**相关的几件事：**编辑器模板**、**AI 填写说明**、**本地钩子拦截**、**用 Python 从提交生成发布说明**。四者互补，不是同一环节重复做四遍。

---

## 推荐工作流（实操顺序）

### A. 每个克隆只做一次（环境）

1. 在仓库根目录执行 **`scripts/setup-git-hooks.cmd`**（或 `.ps1` / `.sh`），或手动：
   - `git config core.hooksPath .githooks`
   - `git config commit.template .gitmessage`（存在 `.gitmessage` 时）
2. 确认：`git config --get core.hooksPath`、`git config --get commit.template` 有输出。

### B. 每次提交（建议按序做）

1. **`git add` 后先看暂存路径**（与标签强相关）  
   - 终端：`git diff --cached --name-only`  
   - 或 IDE 暂存列表。对照 **`.copilot-commit-message-instructions.md`** 里「路径 → 宏」判断需要哪些 **`[XXX_VER]`**。

2. **你定「草稿格式 / 骨架」**（减少 AI 乱格式）  
   - **推荐**用 **`git commit`**（不要加 `-m`）打开编辑器，会载入 **`.gitmessage`**。  
   - 在模板**顶部空白处**先写好：  
     - 第一行：**标题**（可选 **`docs/Git提交类型前缀说明.md`** 中的 `feat:` / `fix:` / `docs:` / `chore:`）；  
     - 空一行；  
     - **正文**：按路径写好 **`[XXX_VER]`** 行（可多段，或同一行用 **`；`** 分隔多段），说明处可先写 **`…`** 或极简词，留给下一步补全。

3. **（可选）让 AI 只「填肉」**  
   - 把当前草稿（含你已写好的 **`[XXX_VER]`**）贴到 Cursor，并明确指令，例如：  
     **「保留所有 `[XXX_VER]` 宏名与分段结构不要改；只把说明写成通顺简体中文，要点与当前暂存改动一致，不要写未改动模块。」**  
   - 将 AI 输出**贴回**提交编辑器，**你再扫一眼** diff 与宏是否仍对。

4. **提交前自检（10 秒）**  
   - 暂存里**有** `work_station/`、`agreement/qtuple/` 等 → 说明里是否**有**对应 **`[XXX_VER]`**？  
   - 暂存**只有** `docs/`、`.githooks`、`.copilot` 等 → 是否**没有**任何 **`[XXX_VER]`**？  
   - 是否整段英文、宏名是否 **`_VER` 结尾**（避免 `FREE_VR`）？

5. **保存关闭编辑器 → 提交**  
   - **`commit-msg` 钩子**会跑 **`docs/validate_commit_message.py`**；失败时看终端 **stderr**，改说明后重新提交。  
   - 非必要不要用 **`--no-verify`**。

### C. 发版 / 出说明前（与单次提交分开）

1. 在仓库根目录执行 **`py docs/生成版本说明.py`**（时间范围在脚本内配置）。  
2. 检查 **`docs/上位机版本发布.md`** 新增块；无 **`[XXX_VER]`** 的提交不会进各工站括号，属预期。

### 工作流小结（谁负责什么）

| 步骤 | 谁为主 | 作用 |
|------|--------|------|
| 路径与 `[XXX_VER]` 骨架 | **你** | 格式与发布脚本对齐，少漏标签 |
| 润色说明句 | **AI（可选）** | 省打字，须在约束下修改 |
| 合法性与路径匹配 | **钩子** | 自动拦截一批错误 |
| 进「上位机版本发布」 | **`生成版本说明.py`** | 事后汇总，依赖历史提交正文 |

## 1. 提交信息模板

- **作用**：执行 `git commit`（打开编辑器）时，预先载入一段提示，减少漏写、格式不统一。
- **本仓库文件**：根目录 **`.gitmessage`**
- **配置**（每个克隆一次，或与钩子一起跑 setup）：
  ```text
  git config commit.template .gitmessage
  ```
  或使用 **`scripts/setup-git-hooks.ps1` / `setup-git-hooks.sh` / `setup-git-hooks.cmd`**，会在存在 `.gitmessage` 时自动设置。
- **注意**：在 IDE 里**只用输入框、不打开编辑器**提交时，**不会**加载该模板；模板主要服务命令行 `git commit`。

---

## 2. 提交信息 AI（Copilot / Cursor）

- **作用**：根据 **diff / 暂存** 补全或润色说明；**推荐**在你已用 **`.gitmessage`** 写好 **`[XXX_VER]` 骨架**后，让 AI **只填说明句、不改标签与结构**（见上文「推荐工作流」B-3）。
- **规则文件**：仓库根目录 **`.copilot-commit-message-instructions.md`**（与 GitHub Copilot 提交生成、本仓库约定对齐）。
- **要点**：简体中文、`[XXX_VER]` 与路径对应、纯文档/脚本路径不要滥写宏等，均以该文件为准。
- **关系**：AI 仍可能写偏；**Hook** 在提交瞬间再拦一道（见第 3 节）。

---

## AI 生成说明的常见痛点与对策

大模型**不保证**一次就符合本仓库约定，常见有：

| 问题 | 说明 |
|------|------|
| **格式** | 整段英文、漏写 `[XXX_VER]`、宏名笔误（如 `FREE_VR`）、纯文档/脚本却写 `[FREE_VER] 三元组` 等 |
| **内容** | 写了未改动的模块（幻觉）、工站与标签不对应、把「校验/脚本」写成「三元组业务」 |

**已有手段（建议叠用）**

1. **规则文件**：**`.copilot-commit-message-instructions.md`** — 改这里会影响 Copilot 生成方向；发现重复出现的错法可往里面加**反例 + 正确示例**。
2. **提交前人工改一版**：把 AI 输出当**草稿**，对照 **「暂存区文件路径」** 改标题/正文；不要未读就点提交。
3. **本地钩子**：**`validate_commit_message.py`** 能拦一部分（非法宏、路径与标签不匹配、纯 docs/ 元信息却写 `[XXX_VER]` 等），**拦不住**「描述与 diff 语义不符但标签形式正确」类问题，仍靠人眼看。
4. **用编辑器提交**：需要模板时，用 **`git commit` 打开带 `.gitmessage` 的编辑器**，比仅在侧栏输入框里生成更少漏项（模板里有类型与 `[XXX_VER]` 提示）。
5. **少用 `--no-verify`**：只有确知钩子误报或紧急推送时才跳过，否则格式/标签错误会直接进入历史，**`生成版本说明.py` 也会扫到脏说明**。

**结论**：AI 适合**起稿或在你给定骨架后填句**；**格式与事实**以你的骨架 + 修改 + 钩子 + `.copilot-commit-message-instructions.md` 为准。若某种错误反复出现，优先**补一条提示词反例**，再考虑是否能在 **`validate_commit_message.py`** 里加规则（可维护成本会上升）。

## 3. 提交内容：Hook 脚本拦截

- **作用**：在 `git commit` **落盘前**检查提交说明是否满足约定（宏名合法、路径与标签是否匹配、纯文档暂存是否误写 `[XXX_VER]` 等）。
- **实现**：**`.githooks/commit-msg`** → 调用 **`docs/validate_commit_message.py`**
- **启用**：`git config core.hooksPath .githooks`（同上，可用 setup 脚本一次配置）。
- **跳过**（仅紧急）：`git commit --no-verify` 或环境变量 `SKIP_COMMIT_MSG_VALIDATE=1`。
- **详情**：**`docs/上位机版本信息生成说明.md`** 中「提交信息校验」小节。

---

## 4. 提交信息：Python 读取 Git 生成发布说明

- **作用**：从 **`git log`** 里按时间范围扫提交说明，解析其中的 **`[XXX_VER]`**，生成 / 追加 **`docs/上位机版本发布.md`** 中的「上位机版本」段落（各工站行后的括号内容）。
- **脚本**：**`docs/生成版本说明.py`**
- **宏定义来源**：**`my_set/AbIni.h`** 中的 `#define XXX_VER`
- **执行**：在仓库根目录执行 `py docs/生成版本说明.py`（或 `python docs/生成版本说明.py`）。
- **说明**：这是**发布前/出包前**的步骤，与 hook「是否允许本次 commit」是两条线；没有写进提交里的 `[XXX_VER]` 不会出现在对应工站行。

---

## 与「类型前缀」的关系

若标题使用 **feat / fix / docs / chore** 等前缀，见 **`docs/Git提交类型前缀说明.md`**。类型前缀概括改动性质；**`[XXX_VER]`** 仍按 **`.copilot-commit-message-instructions.md`** 与钩子逻辑处理。
