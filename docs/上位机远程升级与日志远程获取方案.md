# 上位机远程升级与日志远程获取方案

## 目标

为当前 Qt 上位机增加两个远程运维能力：

1. **远程升级上位机程序**：上位机启动或手动点击时检查服务器版本，下载新版本安装包/可执行文件，校验成功后完成替换并重启。
2. **远程获取日志**：服务端可按设备、工位、日期下发日志获取任务，上位机打包本地日志并上传，服务端返回下载地址或保存到对象存储。

当前工程已经具备 `Qt Network` 能力，`new_production.pro` 中已有 `QT += network`；项目里也已有 HTTP 下载、multipart 上传、三元组 REST 请求和本地日志落盘逻辑，可以在这些基础上扩展。

## 当前工程基础

### 已有网络能力

- `mainlogic.cpp` 中已有 `MainWindow::checkAndUpdateFile()`，当前通过 `QNetworkAccessManager::get()` 从固定地址 `http://163.177.79.53:16888/versions/` 拉取远端目录并下载新的 `new_production_*.exe`。
- `mainlogic.cpp` 中已有 `MainWindow::sendPicture()`，使用 `QHttpMultiPart` + `QNetworkAccessManager::post()` 上传文件，可作为日志压缩包上传的参考实现。
- `agreement/qtuple/qtupleservice.cpp` 中已有 REST GET/POST、请求头、Basic Auth、`curl.exe` 兜底等实现，可作为更稳定的 HTTPS 请求封装参考。
- Windows HTTPS 部署要注意 Qt 5.15 对 OpenSSL DLL 的依赖，当前运行目录已有 `libssl-1_1-x64.dll` 相关文件。

### 已有日志目录

上位机日志主要写入运行目录下：

```text
所有log/
  上位机log/
  设备log/
  dongle的log/
  设备黑盒的log/
  治具log/
  ota升级压测/
```

关键来源：

- `main.cpp` 的 `customMessageHandler()` 写入 `所有log/上位机log/`。
- `qlog/qlog.cpp` 的 `Qlog::save_brush_log()` 写入 `所有log/设备log/`。
- 部分工站、dongle、治具和 OTA 压测日志在各自模块中写入 `所有log/` 的子目录。

## 推荐总体架构

推荐采用“上位机主动轮询服务端任务”的方式，不建议让服务端直接连接产线电脑。

原因：

- 产线电脑通常在内网或 NAT 后面，服务端无法稳定主动访问。
- 上位机主动访问 HTTPS 接口，更容易通过防火墙和代理。
- 权限、安全、审计和失败重试都更好控制。

整体流程：

```text
上位机启动/定时轮询
  -> 上报本机信息和当前版本
  -> 服务端返回待执行任务
     -> remote_upgrade：下载升级包、校验、调用独立升级器替换
     -> fetch_log：选择日志、打包、上传、回报结果
  -> 服务端记录执行状态
```

## 服务端接口设计

### 1. 上位机注册/心跳

用于服务端知道每台上位机的身份、版本、工位和在线状态。

```http
POST /api/factory-tool/heartbeat
Content-Type: application/json
Authorization: Basic xxx 或 Bearer xxx
```

请求示例：

```json
{
  "deviceId": "PC-20260514-01",
  "hostName": "DESKTOP-XXX",
  "station": "freework",
  "appName": "new_production",
  "appVersion": "20260514",
  "toolVersion": "DEBUG_VER...",
  "ip": "192.168.1.10",
  "timestamp": "2026-05-14T14:50:00+08:00"
}
```

响应示例：

```json
{
  "code": 0,
  "message": "ok",
  "serverTime": "2026-05-14T14:50:00+08:00"
}
```

### 2. 查询任务

上位机定时查询是否有远程升级或日志获取任务。

```http
GET /api/factory-tool/tasks?deviceId=PC-20260514-01&appVersion=20260514
```

响应示例：

```json
{
  "code": 0,
  "tasks": [
    {
      "taskId": "task-001",
      "type": "remote_upgrade",
      "priority": 10,
      "payload": {
        "version": "20260515",
        "downloadUrl": "https://server/packages/new_production_20260515.zip",
        "sha256": "xxxx",
        "force": false,
        "silent": false,
        "minBatteryOrIdleRequired": true
      }
    },
    {
      "taskId": "task-002",
      "type": "fetch_log",
      "priority": 5,
      "payload": {
        "dateFrom": "2026-05-14",
        "dateTo": "2026-05-14",
        "logTypes": ["upper", "device", "dongle", "fixture", "ota"],
        "sn": "",
        "maxSizeMB": 200,
        "uploadUrl": "https://server/api/factory-tool/logs/upload"
      }
    }
  ]
}
```

### 3. 回报任务状态

```http
POST /api/factory-tool/tasks/{taskId}/status
Content-Type: application/json
```

请求示例：

```json
{
  "deviceId": "PC-20260514-01",
  "status": "running",
  "progress": 35,
  "message": "正在下载升级包",
  "timestamp": "2026-05-14T14:51:00+08:00"
}
```

状态建议：

- `pending`：已收到，等待执行。
- `running`：执行中。
- `success`：执行成功。
- `failed`：执行失败。
- `rejected`：上位机拒绝执行，例如正在测试中、版本不匹配、校验失败。

### 4. 上传日志

建议使用 multipart 文件上传，方便兼容当前 `sendPicture()` 的实现模式。

```http
POST /api/factory-tool/logs/upload
Content-Type: multipart/form-data
```

字段建议：

| 字段 | 说明 |
| --- | --- |
| `deviceId` | 上位机唯一 ID |
| `taskId` | 日志获取任务 ID |
| `station` | 工站名称 |
| `dateFrom` | 日志开始日期 |
| `dateTo` | 日志结束日期 |
| `file` | zip/7z 压缩包 |
| `sha256` | 压缩包 SHA256 |

响应示例：

```json
{
  "code": 0,
  "message": "ok",
  "logUrl": "https://server/logs/task-002.zip"
}
```

## 上位机配置项

建议新增配置统一放到 `上位机设置.ini`，通过 `SETTINGS` 读取，并在 `agreement/qset/qsetting` 的设置页补齐加载和保存。

建议键名：

```ini
[RemoteOps]
Enabled=true
BaseUrl=https://server/api/factory-tool
AuthUser=factory-tool
AuthPassword=******
DeviceId=PC-20260514-01
Station=freework
HeartbeatIntervalSec=60
TaskPollIntervalSec=30
UseCurl=true

[RemoteUpgrade]
Enabled=true
CheckOnStartup=true
AllowSilentInstall=false
PackageDir=upgrade/packages
BackupDir=upgrade/backup
UpdaterExe=updater.exe

[RemoteLog]
Enabled=true
TempDir=upload/logs
MaxPackageSizeMB=200
KeepTempPackageDays=7
```

注意：

- 当前项目已有 `Tuple/BaseUrl`、`Tuple/AuthUser`、`Tuple/AuthPassword`、`Tuple/UseCurl` 的思路；远程运维可以独立使用 `RemoteOps/*`，避免和三元组业务耦合。
- 密码或 token 不建议明文长期保存；如果现场要求更高，可以使用 Windows DPAPI 做本机加密。
- 新增 `.cpp`、`.h`、`.ui`、`.md` 等文件按项目规则保存为 UTF-8 with BOM。

## 上位机代码改造建议

### 1. 新增远程运维服务类

建议新增独立类，例如：


## 远程升级实现方案

### 推荐升级包格式

不要只下载单个 exe，建议使用 zip/7z 包：

```text
new_production_20260515.zip
  new_production_20260515.exe
  Qt5Core.dll
  Qt5Network.dll
  platforms/qwindows.dll
  libssl-1_1-x64.dll
  libcrypto-1_1-x64.dll
  manifest.json
```

`manifest.json` 示例：

```json
{
  "appName": "new_production",
  "version": "20260515",
  "mainExe": "new_production_20260515.exe",
  "sha256": {
    "new_production_20260515.exe": "xxxx"
  },
  "minVersion": "20250501",
  "releaseNote": "修复远程日志上传，优化升级检查"
}
```

### 升级流程

1. 上位机查询任务或手动检查版本。
2. 服务端返回 `remote_upgrade` 任务。
3. 上位机判断当前是否可以升级：
   - 没有正在测试。
   - 没有关键串口/治具流程运行中。
   - 当前版本低于目标版本。
   - 用户确认升级，或配置允许静默升级。
4. 下载升级包到 `upgrade/packages/`。
5. 校验 SHA256，不通过则删除文件并回报失败。
6. 解压到临时目录 `upgrade/staging/版本号/`。
7. 启动独立升级器 `updater.exe`，把当前进程 PID、目标目录、备份目录、主 exe 名称传给升级器。
8. 当前上位机退出。
9. `updater.exe` 等待主进程退出后备份旧文件、复制新文件、启动新 exe。
10. 新版本启动后上报 `upgrade_success`。

### 为什么需要独立升级器

Windows 下正在运行的 exe 和 DLL 通常无法被当前进程稳定覆盖。当前工程已有下载新 exe 的能力，但真正替换运行目录时建议使用独立升级器或批处理脚本完成：

```text
主程序下载并校验包
  -> 启动 updater.exe
  -> 主程序退出
  -> updater.exe 替换文件
  -> updater.exe 重启主程序
```

升级器要做的事情：

- 等待旧进程退出。
- 备份旧版本到 `upgrade/backup/版本号/`。
- 复制新版本文件到运行目录。
- 失败时回滚备份。
- 启动新版本主程序。
- 写入升级日志 `所有log/上位机log/升级日志_日期.txt`。

### 校验与安全

最低要求：

- 服务端返回文件大小和 SHA256。
- 上位机下载后必须校验 SHA256。
- HTTPS 部署正常，或使用 `curl.exe` 并校验证书。
- 升级包命名带版本号，不覆盖下载历史。

更高要求：

- 对 `manifest.json` 做签名，例如 RSA 签名。
- 上位机内置公钥，只信任签名正确的升级包。
- 服务端按设备 ID、工站、灰度比例下发升级任务。

## 日志远程获取实现方案

### 推荐方式

建议使用“服务端下发任务，上位机主动上传日志包”的方式，而不是在上位机开 HTTP 服务等待服务端下载。

流程：

1. 服务端创建 `fetch_log` 任务，指定设备、日期范围、日志类型。
2. 上位机轮询到任务。
3. 上位机按日期和类型筛选本地 `所有log/` 文件。
4. 上位机把匹配文件复制到临时目录。
5. 生成清单 `manifest.json`，记录文件名、大小、SHA256。
6. 压缩为 zip/7z。
7. 使用 multipart 上传。
8. 上传成功后回报 `logUrl`。
9. 清理超过保留天数的临时压缩包。

### 日志类型映射

建议服务端使用固定类型，上位机内部映射到目录：

| 类型 | 本地目录 |
| --- | --- |
| `upper` | `所有log/上位机log/` |
| `device` | `所有log/设备log/` |
| `dongle` | `所有log/dongle的log/` |
| `blackbox` | `所有log/设备黑盒的log/` |
| `fixture` | `所有log/治具log/` |
| `ota` | `所有log/ota升级压测/` |
| `crash` | `所有log/` 下的 `*_闪退记录.dmp` |

### 打包建议

Qt 5.15 本身没有直接可用的 zip 打包 API，推荐三选一：

1. 随软件部署 `7z.exe`，上位机用 `QProcess` 调用压缩。
2. 引入轻量 miniz/minizip 库，直接在 C++ 内打包。
3. 如果服务器在同局域网并可接受多文件上传，也可以先不压缩，逐个 multipart 上传。

生产现场更推荐 `7z.exe` 或 minizip。压缩包命名示例：

```text
logs_DESKTOP-XXX_freework_2026-05-14_task-002.zip
```

包内结构示例：

```text
manifest.json
upper/xxx_上位机日志_default_2026-05-14.txt
device/1.log
dongle/xxx.log
crash/2026-05-14_10-20-30_闪退记录.dmp
```

### 日志筛选规则

建议按以下顺序筛选：

1. 先按 `dateFrom/dateTo` 匹配文件修改时间。
2. 如果任务带 SN/MAC，再按文件名或文件内容做二次筛选。
3. 如果包大小超过 `RemoteLog/MaxPackageSizeMB`，优先保留上位机日志、崩溃 dump、当天最近文件；其余文件标记为跳过并写入 `manifest.json`。
4. 所有跳过原因都回报给服务端，避免远程排查时误以为日志不存在。

## 失败处理

### 升级失败

常见失败和处理：

| 失败点 | 处理 |
| --- | --- |
| 下载失败 | 重试 3 次，回报失败原因 |
| SHA256 不一致 | 删除包，禁止升级，回报校验失败 |
| 解压失败 | 保留包和错误日志，回报失败 |
| 替换失败 | 升级器回滚旧版本 |
| 新版本启动失败 | 升级器尝试启动旧版本，回报失败 |

### 日志上传失败

处理建议：

- 网络失败时保留压缩包，下一轮继续上传。
- 服务端返回 4xx 时不无限重试，直接回报失败原因。
- 服务端返回 5xx 或超时时可重试。
- 上传完成后再清理临时文件，避免中途失败导致日志丢失。

## 部署注意事项

- 运行目录需要带上 `libssl-1_1-x64.dll`、`libcrypto-1_1-x64.dll`，否则 Qt HTTPS 可能不可用。
- 如果启用 `curl.exe` 兜底，需要随包部署 `curl.exe`，并确认现场电脑能执行。
- 日志和升级包目录要有写权限，建议都放在上位机运行目录下。
- 升级前要确认没有测试流程正在运行，避免升级中断产线测试。
- 服务端必须记录每个任务的执行日志，方便追溯。
- 上位机本地也要写远程运维日志，网络失败时仍能现场排查。

## 最小可落地版本

如果想先快速上线，推荐先做以下最小闭环：

1. 配置 `RemoteOps/BaseUrl` 和认证信息。
2. 手动菜单“上传今日日志”。
3. 服务端提供 multipart 上传接口。
4. 手动菜单“检查升级”，下载 zip 并校验 SHA256。
5. 先弹窗提示用户关闭软件后手动替换，确认稳定后再做自动升级器。

这样风险最低，也能尽快解决远程拿日志和远程发版本的问题。
