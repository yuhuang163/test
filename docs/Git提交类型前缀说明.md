# Git 提交信息类型前缀说明

在写提交**标题**（第一行）时，可在最前面加上 **类型前缀**，便于浏览历史、生成变更日志时快速分类。前缀与本仓库 **`[XXX_VER]`** 等约定可同时使用，互不替代：前者概括「改动的性质」，后者对应「发布说明 / 工站版本行」（详见根目录 **`.copilot-commit-message-instructions.md`**）。

## 常用前缀（推荐）

| 前缀 | 含义 | 典型场景 |
|------|------|----------|
| **feat** | 新增功能 | 新工站能力、新界面流程、新协议能力等 |
| **fix** | 修复缺陷 | 崩溃、错误结果、边界条件、协议/时序问题等 |
| **docs** | 文档更新 | `README`、`docs/*.md`、注释性说明、与代码行为无关的纯文档 |
| **chore** | 构建或辅助工具变更 | 脚本、钩子、生成工具、依赖/工程配置、格式化与无关业务的杂项维护 |

## 标题格式建议

- **形式**：`类型: 简短描述` 或 `类型：简短描述`（中英文冒号均可，团队内统一即可）。  
  **示例**：`feat: 摄像工站增加 CSV 导出路径配置`  
  **与工站标签并存时**：第一行可写 `feat: …`，正文另起行写 `[CAMERA_VER] …`；或按项目习惯把 `[XXX_VER]` 放在标题/正文，但**不要**用类型前缀代替 `[XXX_VER]`（发布脚本只认宏标签行）。
- **长度**：标题宜短（约 50 字内），细节放在正文。

## 与本仓库其它约定的关系

- **有业务代码 / 工站 / 协议改动**：仍须遵守 **`[XXX_VER]`**、路径与标签对照、纯文档路径不要滥写宏等规则（见 **`.copilot-commit-message-instructions.md`**）。
- **仅文档 / 仅工具脚本**（如只改 `docs/生成版本说明.py`、`.githooks`）：可用 **`docs:`** 或 **`chore:`** 作为标题前缀，且**不要**写 `[XXX_VER]`（见该文档「仅改工具脚本 / 校验 / 钩子」一节）。

## 可选扩展（需要时再采用）

若团队希望更细分类，可约定补充例如：`refactor`（重构不改行为）、`style`（格式/命名）、`test`（测试相关）等；未统一前建议仍用上表四类，避免前缀过多难以执行。
