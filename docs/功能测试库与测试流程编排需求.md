# 功能测试库与测试流程编排 — 需求说明

> 本文档用于 Review 与后续开发对齐。实现时须遵守仓库 `.cursor/rules/qt-cpp-project.mdc`（`SETTINGS`、分层、`UTF-8 with BOM` 等）。

## 一、背景与目标

**项目**：`new_product_test`（Qt 5.15 / MSVC / C++17，配置走 `SETTINGS` + `上位机设置.ini`）。

**现状参考**：

- `advance/imagewindow/draggablecheckbox.*`：可拖拽 `QCheckBox`，用于 `platform/settings/qsetting` 自由工站「测试顺序」编排（`TestOrder_*` 写入 `上位机设置.ini`）。
- `work_station/freework/qfreework.cpp`：按 `TestOrderMeta/SelectedStation` 与 `TestOrder/<工站>` 加载步骤索引执行。
- 平台层：`platform/serial/serial_channel`、`platform/settings/qsetting`；协议层 `agreement/`；工站 `work_station/`。

**目标**：为「一拖多 / 可配置功能块」做准备，将**单条测试用例**与**各工站测试流程**外置为 ini（统一放在 `exe 同级/test_case/`），并在**上位机设置**中新增独立 Tab 页完成可视化配置。**总流程不再写入 `上位机设置.ini`**；旧 `TestOrder_*` 不自动迁移，由新 Tab 人工重建流程。

## 二、模块 A：功能测试库（外置 ini，单 case 一条）

### 2.1 新建 UI 组件（类比 `DraggableCheckBox`）

新建可拖拽测试项块组件（建议名：`TestCaseBlock` 或 `DraggableTestCaseBlock`），继承或组合 `QCheckBox` + 拖拽能力，**行为上参考** `DraggableCheckBox`，但职责是「一条可配置的功能测试 case」，而非仅索引勾选。

**必须支持**：

| # | 功能 | 说明 |
|---|------|------|
| 1 | 右键菜单 | 「打开设置」「从流程移除」；**仅从编排区移除，不删除** `test_case/` 下磁盘 ini（不提供「删磁盘文件」入口） |
| 2 | 发送关键参数 | **声明式（方案 B）**：UI 选择 `DeviceCmd` 及参数，运行时调用 `QProtocolManager::current()->set/get(DeviceCmd, QVariant)`（与 FCTP 等协议枚举一致）；禁止 case 内自定义二进制组帧 |
| 3 | 延时 | 「开始测试前延时 ms」「结束测试后延时 ms」 |
| 4 | 卡控 | 开关「是否卡控」；若开启，卡控项可配置（上下限、比较符、期望值来源等） |
| 5 | 保存与命名 | 保存校验通过后：① 复选框/块显示文字更新为**测试项名称**；② 在 `exe 同级目录/test_case/<测试项名称>.ini` 生成/覆盖该 case 的 ini |
| 6 | 保存前校验 | 未通过自动检测则禁止保存，弹窗/日志提示具体错误字段，引导修改 |
| 7 | 钩子 | 支持「外部引用钩子函数」：ini 中记录钩子标识（如 `hookId` 或 `dll/export` 名，具体方案需与 `platform/test_case` / 自由工站钩子扩展方式一致）；常规流程走统一执行器，仅特殊 case 走钩子 |

**卡控数据类型约束（硬性）**：

- 底层上报用于卡控的字段必须来自**统一的上报类型注册表**（与 `agreement/qProtocol/qprotocol_types.h` 中 `Protocol*Data` / `ProtocolManager` 信号体系对齐）。
- UI 中「卡控内容」下拉/表单只能选择注册表已登记的类型 + 字段，**禁止** per-case 自由填字段名。
- **首版卡控字段子集：待定**（实现前与协议/工站同事确认后写入注册表；P0/P1 可先做空注册表 + 扩展接口，校验器对未登记类型报错）。

### 2.2 Case 配置对话框（UI 表单 → ini）

- 右键「打开设置」弹出**独立配置对话框**，所有字段表单化，禁止手改 ini。
- 字段分组：`[Meta]`、`[Send]`、`[Timing]`、`[Gate]`、`[Hook]`（可选）。

**`[Send]` 声明式（方案 B，已拍板）**：

- **动作类型**：`Set` 或 `Get`（对应 `protocolManager.current()->set(cmd, var)` / `get(cmd, var)`）。
- **命令**：`DeviceCmd` 枚举项（下拉，数据源与 `qprotocol_types.h` 中 `DeviceCmd` 及当前协议实现一致；FCTP 专项参数按 `Qfctp::set/get` 已支持的 `QVariant` 形态配置）。
- **参数**：按所选 `DeviceCmd` 动态展示表单项（如 `FacResult`、`Sn`、`ProtocolUInt32ValueData` 等），序列化进 ini；保存时校验类型与必填项。
- 执行端统一经 `QProtocolManager`，不按 case 写死 `Qpb`/`Qfctp` 分支。

**case ini 示例（片段）**：

```ini
[Meta]
Name=EnterFacMode
DisplayName=进入工厂模式

[Send]
Action=Set
DeviceCmd=FacMode
; 无参或 Variant 序列化字段，如 VariantType=empty

[Timing]
DelayBeforeMs=100
DelayAfterMs=500

[Gate]
Enabled=true
ReportType=ProtocolRssiData
Field=rssi
Op=range
; 上下限可引用 SETTINGS 键或 ini 内字面量

[Hook]
Enabled=false
HookId=
```

- 「保存」按钮：先跑校验器 → 通过则写 `test_case/<Name>.ini` → 更新块标题。

**case 命名（已拍板）**：

- 仅允许**操作系统文件名合法字符**（建议：`[A-Za-z0-9_-]+`，禁止 `\ / : * ? " < > |` 及空格、中文）；`[Meta]/Name` 与文件名一致。
- **保留名禁止**作为 case 名：`总的测试流程`（及与 `总的测试流程.ini` 冲突的名称）。

**校验器（保存前自动检测）至少包含**：

- 测试项名称非空、符合文件名规则、与 `test_case/` 下其它 ini **不重名**（见 2.3）。
- `[Send]`：`DeviceCmd` 合法、该 Cmd 下必填 `QVariant` 齐全、`Set/Get` 与协议能力匹配。
- 延时 ≥ 0。
- 卡控开启时：卡控类型已选、上下限/比较符合法、与统一上报类型匹配。
- 钩子开启时：钩子 id 在注册表中存在或标记为「待实现」并警告。

### 2.3 文件布局（与 exe 同级）

```text
<exe目录>/
├── 上位机设置.ini          ← 仅保留全局参数（阈值、MES、串口等），**不含**测试流程编排
└── test_case/
    ├── 总的测试流程.ini    ← 各工站的测试顺序（模块 B），唯一总流程文件
    ├── 功能A.ini           ← 单条测试 case（模块 A）
    ├── 功能B.ini
    └── ...
```

**目录约定**：

- `test_case/` 下**禁止同名文件**（含 `总的测试流程.ini` 与各 case ini），保存时自动去重/报错。
- 每保存成功一个 case → **立即**生成/更新 `test_case/<测试项名称>.ini`。
- 流程编排保存 → 更新 `test_case/总的测试流程.ini`（不按工站拆多个总流程文件）。

**`总的测试流程.ini` 结构建议**（示例，实现时可微调键名）：

```ini
[Meta]
Version=1
; 与 上位机设置.ini [TestOrderMeta] 同源语义（仅 UI 记忆时可写，流程数据以 Station 段为准）
SelectedStation=SUCTION_TEST
SelectedStationName=吸力测试工站

[Station/SUCTION_TEST]
Items=EnterFacMode,ReadRssi,WriteSn    ; case 名 = test_case 下文件名（无扩展名），逗号分隔

[Station/FREE_WORK]
Items=...
```

- 单 case 参数仍在各自的 `test_case/<名称>.ini`，**不**写入 `总的测试流程.ini`。
- 若改名导致旧 case 文件残留，需定义策略（提示手动删 / 保存时询问是否删除旧名文件）。

### 2.4 在上位机设置中的位置（已拍板）

- **`platform/settings/qsetting.ui` 另起一个新 Tab**（如「测试流程编排」），与现有「自由工站 TestOrder」等旧页**并存**；旧 Tab 不删，新系统逐步替代执行端读 ini 的路径。
- 新 Tab 内以 **测试流程页** 为主（见 3.2 UI 布局）；case 详细参数在块右键「打开设置」对话框中编辑。

## 三、模块 B：测试流程编排（`test_case/总的测试流程.ini`）

### 3.1 与现有工站配置的关系

- **总流程唯一落点**：`test_case/总的测试流程.ini`，内按工站分段 `[Station/<工站键>]` 存放 case 名称顺序。
- **工站键与显示名（已拍板）**：与 `上位机设置.ini` 中 `[TestOrderMeta]` 一致，例如 `SelectedStation=SUCTION_TEST`、`SelectedStationName=吸力测试工站`；`总的测试流程.ini` 的 section 使用 **`SelectedStation` 的键值**（如 `SUCTION_TEST`），显示名写入 `[Meta]` 或工站段注释/UI 专用键，供下拉框展示。
- **`上位机设置.ini` 不再保存测试流程列表**；旧 `TestOrder_*` 不再写入、不自动导入新 ini。`TestOrderMeta/SelectedStation` 可保留为「设置页当前选中工站」的 UI 状态（与现网一致）。
- **执行范围（已拍板）**：架构按**后期覆盖所有工站**设计；实现分期可先接 `qfreework`，其它工站后续接入同一 `TestCaseRunner`。
- 打开设置页 / 切换工站时：从 `总的测试流程.ini` 读取对应工站 `Items`，**自动恢复**编排区排布。
- **不自动迁移旧配置**：不使用 `FREEWORK_TEST_LIST` / `testFunction.cpp` 的 `name` 将 `TestOrder` 整型索引映射为 case 名；旧 `上位机设置.ini` 的 `TestOrder_*` 仅人工对照，在新 Tab 中手动新建 case 与流程（可选：后续单独做「导入向导」，本需求不包含自动映射）。
- 编排区块上显示文字为**具体测试项名字**（与 `test_case/<名称>.ini` 文件名或 `[Meta]/Name` 一致），不再使用数字索引。

### 3.2 流程编排 UI（新 Tab — 测试流程页，已拍板）

**整体布局**：

- 新 Tab 内为**测试流程页**：上方工站选择 + 工具按钮；中部 **`QScrollArea`** 作为可滚动编排区；块在 ScrollArea 内横向/流式排列，支持拖拽排序。
- 与旧「自由工站 TestOrder」拖拽区**独立**，不复用旧 `DraggableCheckBox` 索引逻辑。

| 元素 | 行为 |
|------|------|
| 工站选择 | 下拉：项的 `data` = `SelectedStation` 键（如 `SUCTION_TEST`），显示文本 = `SelectedStationName`（如 `吸力测试工站`）；切换后从 `总的测试流程.ini` 加载 `[Station/<键>]/Items` |
| 编排区 | `QScrollArea` + 内层布局；放置 `TestCaseBlock`（显示 case 名） |
| **「+」** | 在编排区末尾新增**空白**功能块；右键「编辑」填写并保存后生成 `test_case/<名>.ini`，块标题更新 |
| **「新增已有功能块」**（按钮） | 弹出 `test_case/` 下已有 case ini 列表（多选），导入为块并追加到当前工站流程 |
| 右键块 | 打开 case 设置 / **从流程移除**（不删磁盘 ini）/ （可选）打开 ini 所在目录 |
| 一键清空 | 清空当前工站在编排区的块列表（**不删** `test_case/` 下任何 ini） |
| 保存流程 | 写回 `test_case/总的测试流程.ini` 的 `[Station/<SelectedStation>]/Items` |

**流程级选项（已拍板，可放在工站栏或总流程 Meta）**：

- **卡控失败是否结束整单测试**：在流程界面可配置（建议 per 工站或 per case，默认 per case 字段 `StopTestOnGateFail=true/false` 写入 case ini 的 `[Gate]` 或工站段 `StopTestOnGateFail`）；为 `true` 时卡控失败即中止后续 Items 并判 FAIL。

**块与 ini 关系**：

- 每个块绑定 `test_case/<名称>.ini`；块显示名 = case 名（文件名）。
- 空白块**不得**写入 `Items`，须保存 case 成功后再加入流程列表。
- `总的测试流程.ini` 只存各工站 **case 名顺序** + Meta；单 case 的 Send/卡控/延时仍在各自 ini。

### 3.3 执行端（后续迭代，本需求可只定接口）

- `qfreework`（或其它执行工站）读取：`test_case/总的测试流程.ini` 得当前工站顺序 → 按名加载 `test_case/<名>.ini` → 统一发送 + 卡控 + 钩子。
- 需定义运行时接口：`TestCaseRunner::loadFromIni(path)`、`runStep()`、`evaluateGate(Protocol*Data)`。

## 四、非功能与工程约束

- **编码**：新建/修改 `.cpp/.h/.ui` 使用 **UTF-8 with BOM**。
- **配置**：`上位机设置.ini` 仍经 `SETTINGS` 读写全局参数；`test_case/` 下所有 ini（含 `总的测试流程.ini` 与各 case）用 `QSettings::IniFormat` 独立读写，根路径为 `QCoreApplication::applicationDirPath() + "/test_case"`。
- **分层**：
  - UI：`platform/settings/qsetting`
  - 可拖拽块组件：`advance/` 或 `platform/settings/widgets/`
  - **禁止**协议解析写进 UI 类
  - 执行与卡控：`work_station/freework` 或新建 `platform/test_runner/`
- **不要**新建大而全的「操作码对照 md」；类型中文名在现有枚举/注册表内维护。
- **兼容**：执行端优先读 `test_case/总的测试流程.ini`；**不做** `TestOrder_*` → case 名自动导入，**不做** `FREEWORK_TEST_LIST` name 映射；旧流程与新系统并存，由产线在新 Tab 重新配置。

## 五、验收标准（Review 勾选）

### 模块 A

- [ ] 新块组件可拖拽；右键可设置、可从流程移除（磁盘 ini 仍存在）
- [ ] 设置对话框可配置发送参数、前后延时、卡控（类型受统一上报约束）
- [ ] 保存前校验失败有明确报错；成功则块标题更新且 `test_case/<名>.ini` 存在
- [ ] 同名 case 无法保存或提示冲突
- [ ] 钩子字段可配置且文档说明如何注册

### 模块 B

- [ ] 上位机设置新 Tab；工站切换后流程区恢复正确
- [ ] 「+」空白块、导入、拖拽排序、一键清空、右键改功能均可用
- [ ] 流程保存后 `test_case/总的测试流程.ini` 中对应工站为**测试项名称**顺序，重启后仍正确
- [ ] `上位机设置.ini` 中不再新增/依赖测试流程键；删块不删 `test_case/*.ini`
- [ ] 块显示名为 case 名，与 `test_case` 中 ini 一致
