# Agreement 协议层后续深度优化与重构计划

## 目标说明

继 `agreement` 目录下的遗留“上帝类”协议（ADB、Shell、AT、DJI Bulk）完成了基础的“四层物理目录分离”之后，我们成功建立起了“一外设一模块”与“驱动与协议分离”的干净代码结构。
本计划旨在进行“第二阶段”的深入重构：即针对各协议内部的**数据流解耦、CPU与内存开销缩减、以及历史状态机的公共抽象**进行优化。

## 阶段规划

### 阶段 1：治具 PCBA 解析收敛与字段澄清 (P0 优先级)

*   **问题背景**：
    目前在 `agreement/fixture/device/fixture_pcba_device.cpp` 的 `parseFixturePacket` 流程中，依然包含由于历史原因引入的依赖 `SETTINGS` INI 配置节点分支来判断协议版本的旧逻辑。这种隐式环境依赖极易导致产线测试出错。同时，上报的数据结构中的 `shipCurrent` 字段有“船运电流”和“待机电流”的一词多义现象，导致业务层开发容易混淆。
*   **重构方案**：
    1.  **废弃隐式配置分支**：移除对 `SYSTEM/TestAudioCurrent` 等节点配置的判断，强制收敛到定版 V2 协议长包（`length=0x17` 或 `0x18`）的绝对偏移量提取。
    2.  **澄清与重命名数据结构**：在 `fixture_uart_types.h` 中，将具有多重语义的 `shipCurrent` 废弃，拆分为含义清晰的两个变量：`cargoCurrentUa` (船运电流) 与 `standbyCurrentUa` (待机电流)。并在 `test_base` 等工站引用处同步修改。

### 阶段 2：治具多协议过滤与收帧公共化 (P1 优先级)

*   **问题背景**：
    当前 `Fixture_uart` 的硬件串口在收到一段流数据时，会直接并发广播给所有挂载的协议（如 PCBA、压感、IMU、相机）。这造成如果我们在单纯测试 PCBA 板时，后台同时也在无效地跑另外三种协议的帧结构正则匹配，耗费 CPU 开销。此外，各个组件内部各自实现了一套基于 `0x55/0xCC` Magic 首字节校验、长度截断、尾部 `0xAA` 校验的 `RingBuf` 收包状态机。
*   **重构方案**：
    1.  **多协议开关过滤机制**：引入 `FixtureUartProfile`（例如 `PcbaOnly`, `PressOnly` 等），在系统工站初始化时指定 Profile，底层硬件接收到数据时，根据当前激活的 Profile 拦截多余的数据投递。
    2.  **公共取帧层 `RingFrameReader`**：在 `qtransport` 或者公共组件区抽离出一个通用的 `RingFrameReader` 类。通过注入帧格式配置（Magic头，长度偏移，包尾标志），统一切片和校验逻辑，消灭散落各处的相似状态机。

### 阶段 3：双栈延迟加载（懒加载）与瘦身 (P2 优先级)

*   **问题背景**：
    在 `test_base.cpp` 的构造函数中，存在着典型的“全量实例化”问题：不管当前具体运行的是哪个产测流程，系统总是将 `Qpb`, `Qfctp`, `Qaiot`, `Qroot`, `QatManager` 全都一股脑 `new` 出来。这不仅让内存迅速膨胀，而且这些实例还会去并发争抢 Dongle 的物理串口句柄，甚至在后台驻留无用的心跳定时器。
*   **重构方案**：
    1.  **取消全量 `new` 操作**：将 `test_base` 构造函数中固定实例化的代码剥离，指针默认置为 `nullptr`。
    2.  **动态绑定（延迟加载）**：封装 `test_base::initializeProtocol()`。在加载当前工站配置、明确得知即将使用的具体协议（如 `qfctp`）后，再实时将该类进行实例化并挂载到 `DongleSerialPort` 通道上，完成极限瘦身。

### 阶段 4：统一量测数据流与状态机解耦 (长期优化)

*   **问题背景**：
    像 SCPI 或 Modbus 等仪器设备（万用表，程控电源等）向上传递测量值时，其接口的参数类型依然是纯文本的 `QString`，业务层收到数据后还要各自去分辨这是一个电压值还是电流值，解析繁琐。
*   **重构方案**：
    引入面向对象的统一测量模型：`ProtocolMeasureData` 结构体，内部包含传感器类型、测量通道、具体数值（浮点或定点），规范化量测仪器的返回值格式，以此与单纯的状态响应彻底解耦。
