# 三元组数据获取说明

本文档根据当前项目代码整理，说明“产品 Key / 产品 ID / 产品密钥”这组三元组数据在前端流程中是如何获取、缓存并用于烧录的。

## 1. 三元组数据包含什么

代码中三元组字段主要是：

- `productKey`：产品 Key
- `deviceName`：产品 ID
- `deviceSecret`：产品密钥

部分接口会同时返回：

- `sn`：产品 SN
- `status`：烧录/QC 状态
- `availableCount`：可用数量

这些字段最终会写入 `store.state.toolState`，也就是当前工具页面的表单状态。

## 2. 数据来源

三元组不是本地生成的，正常模式下来自后端接口。接口统一封装在 `src/api/tool.ts` 的 `ApiTool` 中。

### 2.0 请求域名、登录和公共请求头

请求域名来自 `src/config/index.ts` 的 `BASE_URL`，通过 `.env.*` 里的 `VITE_ENV_TAG` 选择：

| 环境标记 | 后端域名 |
| --- | --- |
| `dev` | `http://192.168.200.140:8080` |
| `prod` | `https://lute-aiot-mac.luteos.com` |
| `build-dev-inner` | `http://192.168.200.140:8080` |
| `build-dev` | `https://983ug2va5885.vicp.fun` |
| `build-prod` | `https://lute-aiot-mac.luteos.com` |

例如当前 `.env.dev` 中是：

```text
VITE_ENV_TAG=dev
```

所以开发环境下三元组接口完整地址会是：

```text
http://192.168.200.140:8080/api/mac-addresses/applyTupleByMac
http://192.168.200.140:8080/api/mac-addresses/apply-oem-triplet
http://192.168.200.140:8080/api/mac-addresses/apply-wifi-triplet
```

正式打包环境下会变成：

```text
https://lute-aiot-mac.luteos.com/api/mac-addresses/applyTupleByMac
https://lute-aiot-mac.luteos.com/api/mac-addresses/apply-oem-triplet
https://lute-aiot-mac.luteos.com/api/mac-addresses/apply-wifi-triplet
```

所有请求默认带这些请求头，逻辑在 `src/hooks/web/use-axios.ts`：

```http
Content-Type: application/json
Device-Id: <当前机器 deviceId>
APP-Version: <当前应用版本>
Authorization: Basic <base64(username:password)>
```

是否需要登录：需要。登录时前端把用户名和密码拼成 `username:password`，再用 `btoa()` 转成 Base64，缓存为 token。之后三元组接口都会带：

```http
Authorization: Basic <token>
```

登录校验接口是：

```text
GET /api/mac-addresses/auth
```

完整登录地址同样拼 `BASE_URL`，例如开发环境：

```text
http://192.168.200.140:8080/api/mac-addresses/auth
```

登录请求本身没有 body，只靠请求头里的 `Authorization: Basic <base64(username:password)>` 校验。登录成功后，接口返回的 `res.data.authorities` 会作为用户角色缓存。

### 2.1 BLE 流程：通过 MAC 获取三元组

调用方法：

```ts
ApiTool.applyTupleByMac({
  mac,
  sku,
  position
})
```

对应接口：

```text
GET /api/mac-addresses/applyTupleByMac
```

开发环境完整 URL 示例：

```text
http://192.168.200.140:8080/api/mac-addresses/applyTupleByMac?mac=DAC110000C57&sku=xxxxxxx&position=L
```

这个接口是 `GET`，不会发送 JSON body，参数放在 query string 里：

```json
{
  "mac": "DAC110000C57",
  "sku": "xxxxxxx",
  "position": "L"
}
```

实际发送到 URL 时类似：

```text
?mac=DAC110000C57&sku=xxxxxxx&position=L
```

参数含义：

- `mac`：扫码或流程中已有的 MAC
- `sku`：产品 SKU
- `position`：左右标记，根据本地配置或设备读取结果得到

代码里按返回对象直接读取字段，预期响应结构类似：

```json
{
  "success": 0,
  "mac": "DAC110000C57",
  "productKey": "x77MDK",
  "deviceName": "guu05i82k091i0zs",
  "deviceSecret": "dfu5mwe16n7vfm43",
  "sn": "M5P0BP183PUDAC110000C570E57FFN10003",
  "status": 1,
  "availableCount": 10
}
```

代码位置：

- `src/api/tool.ts`
- `src/hooks/web/_fileds/_use-queue-step-event.ts`

### 2.2 WiFi 模组流程：通过批次号 + MAC 获取三元组

调用方法：

```ts
ApiTool.applyTupleByMacBatch({
  mac,
  batchNumber
})
```

对应接口：

```text
POST /api/mac-addresses/apply-oem-triplet
```

开发环境完整 URL 示例：

```text
http://192.168.200.140:8080/api/mac-addresses/apply-oem-triplet
```

这个接口是 `POST`，发送 JSON body：

```json
{
  "mac": "DAC110000C57",
  "batchNumber": "BATCH20260507001"
}
```

参数含义：

- `mac`：设备 MAC，通常来自 WiFi 模组读取步骤
- `batchNumber`：批次号，来自产品配置里的 `batchCode`

代码里从 `res.data` 取三元组，预期响应结构类似：

```json
    "sn": "M5P0BP183PUDAC110000C570E57FFN10003",

```

代码位置：

- `src/api/tool.ts`

## 3. 页面流程如何触发获取

三元组申请不是按钮直接写死在页面里，而是由“步骤队列”驱动。

默认步骤配置在 `src/global/constants/queue-step.ts`：

- BLE QC 流程包含 `ST_APPLY_TUPLE`
- WiFi QC 流程包含 `ST_APPLY_TUPLE_BATCH_SN_MAC` 和 `ST_APPLY_TUPLE_BATCH_MAC`

页面点击开始后，`src/components/ToolPanel/components/ScanCell.vue` 调用 `initStepQueue()`。

之后 `src/hooks/web/use-queue-step-generate.ts` 会按当前步骤队列逐个执行，遇到三元组相关步骤时，会进入 `src/hooks/web/_fileds/_use-queue-step-event.ts` 中的 `StepActionMap`：

- `ST_APPLY_TUPLE` -> `onGetTupleByMac()`
- `ST_APPLY_TUPLE_BATCH_MAC` -> `onGetTupleByMacBatch()`
- `ST_APPLY_TUPLE_BATCH_SN_MAC` -> `onGetTupleBySnMac()`

## 4. 获取成功后保存到哪里

三元组接口返回成功后，会统一调用 `_setTupleData(data)`：

```ts
Object.assign(form.value, {
  productKey,
  deviceName,
  deviceSecret,
  sn,
  status
});
```

这里的 `form.value` 来自：

```ts
const form = computed(() => store.state.toolState);
```

所以数据最终保存到 `store.state.toolState`。

页面上显示的“产品Key / 产品ID / 产品密钥 / 产品SN / 状态”也直接从 `toolState` 读取，位置在 `src/components/ToolPanel/components/ScanCell.vue`。

## 5. 烧录时如何使用三元组

获取三元组后，烧录步骤会从 `toolState` 取值。

### 5.1 BLE 设备烧录

BLE 三元组拆成三条指令写入：

- `W_P_KEY` 写入 `productKey`
- `W_D_NAME` 写入 `deviceName`
- `W_D_SECRET` 写入 `deviceSecret`

烧录队列配置：

```ts
export const QUEUE_WRITE_TUPLE = [
  _O.W_P_KEY,
  _O.W_D_NAME,
  _O.W_D_SECRET,
];
```

发送位置：

- `src/global/constants/queue-write.ts`
- `src/hooks/web/_fileds/_use-com-write-action.ts`

### 5.2 WiFi 模组烧录

WiFi 模组三元组会拼成一个字符串写入：

```ts
`${productKey}${deviceName}${deviceSecret}`
```

对应操作是 `W_MO_TUPLE`。

发送位置：

- `src/global/constants/queue-write.ts`
- `src/hooks/web/_fileds/_use-com-write-wifi-action.ts`

## 6. 演示模式

如果开启演示模式，代码不会请求后端接口，而是从 `MockTuple` 里按 MAC 取模拟数据。

相关代码在：

- `src/config/collection-mock.ts`
- `src/hooks/web/_fileds/_use-queue-step-event.ts`

## 7. 获取失败时的处理

三元组接口请求失败时，会：

- 写入错误日志：`获取设备凭证失败`
- 打开弹窗：`三元组获取异常，请重试`
- 抛出异常，让当前步骤标记为错误

相关逻辑在 `src/hooks/web/_fileds/_use-queue-step-event.ts`。

## 8. 简化链路图

```text
点击开始
  -> ScanCell.vue 调用 initStepQueue()
  -> use-queue-step-generate.ts 按步骤队列执行
  -> 命中 ST_APPLY_TUPLE / ST_APPLY_TUPLE_BATCH_*
  -> _use-queue-step-event.ts 调用 ApiTool 申请三元组
  -> src/api/tool.ts 请求后端接口
  -> 返回 productKey / deviceName / deviceSecret / sn
  -> _setTupleData() 写入 store.state.toolState
  -> 页面展示三元组信息
  -> 烧录步骤从 toolState 取值并通过串口写入设备
```

## 9. 注意点

- 当前仓库只包含前端/Electron 侧逻辑，三元组真正分配规则在后端接口中，仓库里看不到后端如何生成或分配。
- BLE 和 WiFi 的申请接口不同，烧录方式也不同：BLE 分三条写入，WiFi 拼成一条写入。
- `deviceSecret` 在发送日志里会做隐藏处理，避免完整密钥直接显示在日志面板。
