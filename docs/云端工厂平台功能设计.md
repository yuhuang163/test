# 云端工厂平台功能设计

> 面向产测上位机（Qt）与云端后台、网页管理端的整体方案。  
> 基于当前仓库已有能力整理：**日志上传/查询**、**测试数据上报**、**配置下发**、**工厂云登录**、**旧版 exe 自更新**、**test_case ini**、**SETTINGS 阈值** 等（**不含** 各厂 MES 专网过站对接）。  
> **配套文档**：[后端设计](./云端工厂平台-后端设计.md)（**Python / FastAPI**）| [前端设计](./云端工厂平台-前端设计.md)（**Vue 3 / JavaScript**）

## 1. 目标与边界

### 1.1 要建设的 8 项能力

按 **工厂云建设范围** 归纳如下（与下表序号一致）。其中 **日志、测试数据、配置下发、汇报可视化** 为业务侧最常提到的四类，见 **§1.1.1**。


| #   | 能力           | 一句话                                                                 |
| --- | ------------ | ------------------------------------------------------------------- |
| 1   | **日志管理**     | 上位机打包 `所有log` 上传；网页按工厂/设备/时间查询，**在线预览、整包下载**                        |
| 2   | **测试数据**     | 过站/不过站分项 JSON 上报；网页查询、筛选、**导出提取**（**不替代** 各厂 MES 专网接口）              |
| 3   | **配置下发**     | 阈值卡控、测试用例/流程 ini、发布版本等云端维护，上位机拉取并写入 SETTINGS / `test_case`            |
| 4   | **上位机 OTA**  | 检测新版本 exe、下载、校验、替换/重启                                              |
| 5   | **登录与权限**    | 工厂云工站/产线账号、角色、Token、工站授权、登录审计                                     |
| 6   | **网页运营管理**  | 统一门户：维护阈值、用例、上位机版本、日志与测试数据查询（**不替代** 产测工站界面）                      |
| 7   | **统一 API 协议** | REST over HTTPS + JSON；文件类 multipart；统一 BaseUrl、请求头、错误码与版本头          |
| 8   | **汇报与可视化**  | 产测数据沉淀、统计图表、管理汇报；**开源可视化平台（如 Grafana）及汇报模板方案待立项确认**（见 §1.1.1 ④） |


#### 1.1.1 业务视角四类能力（与上表对应关系）

| 业务诉求        | 对应能力 # | 范围说明 |
| ----------- | ----- | ---- |
| ① 日志上报 / 查看 / 下载 | **#1** | 上位机：`log_upload_service` 打包上传；云端：列表、详情预览、zip 下载；按电脑名/工厂筛选 |
| ② 测试数据提取      | **#2** | 上位机：`test_data_upload_service` + 本地 `test_record_store`；云端：`test-records` 入库与网页「测试数据」查询导出；**不上传 CSV 文件、不对接 MES** |

| ④ 汇报可视化与数据库   | **#8**（+#2 数据基础） | **一期**：SQLite/业务库支撑日志、测试数据查询；**二期待定**：指标看板、周/月报模板、是否引入 **Grafana** 或同类开源 BI；需与质量/管理层确认汇报口径后立项 |

- 图表粒度：工站直通率、失败项 TOP、设备/人员维度等  
- **Grafana** vs 自建 Vue 报表 vs 导出 Excel/PPT 为主  
- 与 MES/客户报表的边界（工厂云报表 **不替代** 客户 MES 出货报表）

### 1.2 不在本方案内（保持现状或另立项）

- 设备 BLE/WiFi **固件 OTA**（`mainwindow` / `root_ble_ota2`，走 dongle 与产品协议）
- **各厂 MES 专网接口**（`agreement/qmes/*`，客户定制 HTTP）及 **MES 过站指令/回传**：仍由各厂 MES 模块处理；工厂云 **不代理、不替代** MES 过站协议
- 设备 **云端 License 注册**（`lib/productlicense`，`myusmile.online` 等独立域名）
- **#8 汇报与可视化** 的完整方案（Grafana 部署、大屏、定制汇报 PPT）在 **§1.1.1 ④** 评审通过前 **不纳入当前迭代交付**

## 4. 是否需要统一协议（能力 #7）

**需要。** 建议采用 **REST over HTTPS + JSON**，文件类用 **multipart/form-data**；不另建二进制私有协议。

### 4.1 公共约定

**BaseUrl**：

```ini
[FactoryCloud]
BaseUrl=https://fctp.luteos.com
Environment=prod          ; dev | prod | custom
AuthUser=                 ; 工厂云账号
AuthPassword=
StationKey=FREE_WORK      ; 与 TestOrderMeta/SelectedStation 对齐
DeviceId=                 ; 可选；空则登录时用 QSysInfo::machineHostName()
RememberUsername=true
RememberPassword=false
```

**请求头**（凡 `/api/factory-tool/*` 统一由 `factory_cloud_client` 注入）：

```http
Content-Type: application/json          ; JSON 接口
Device-Id: <当前电脑名，见 §5.6；默认 QSysInfo::machineHostName()>
APP-Version: <上位机 exe 版本，如 FREE_VER 对应版本号>
Station-Key: <工站标识>
Authorization: Bearer <FactoryCloud/Token>   ; 登录后必填，见 §5.6
X-Client-Build: 20260611                 ; 可选，构建日期
```

**统一响应包络**：

```json
{
  "code": 0,
  "message": "ok",
  "data": { }
}
```


| code | 含义               |
| ---- | ---------------- |
| 0    | 成功               |
| 401  | 未登录 / Token 过期   |
| 403  | 无权限              |
| 404  | 资源不存在            |
| 409  | 版本冲突（用例/阈值被他人更新） |
| 5xx  | 服务端错误            |


上位机：`code != 0` 时 `showlog` + 写调试日志，**不静默失败**。

### 4.2 与产品协议（FCTP/QPB）的关系


| 层级              | 协议                     | 用途           |
| --------------- | ---------------------- | ------------ |
| 上位机 ↔ 云端        | **HTTP REST**          | 日志、数据、配置、OTA |
| 上位机 ↔ dongle/产品 | **FCTP/QPB/QROOT**（现有） | 产测指令、设备 OTA  |
| 上位机 ↔ 治具        | **串口 PCBA/压感**（现有）     | 治具控制         |


云端能力 **不应** 塞进 BLE/FCTP TLV；保持「配置与数据走 HTTP，产测走现有协议栈」。

### 4.3 通信方式：JSON 与文件分工

统一走 **HTTPS**；按内容类型分三种，**不另建二进制私有协议**。

```text
┌─────────────────────────────────────────────────────────────┐
│ ① JSON（application/json）                                   │
│    登录、check、manifest、阈值、列表查询、管理端 CRUD            │
│    响应一律 { code, message, data }（文件下载接口除外）          │
├─────────────────────────────────────────────────────────────┤
│ ② 上传文件（multipart/form-data）                              │
│    上位机 → 云：日志 zip                                       │
│    网页 → 云：上位机 exe（管理端发版）                          │
├─────────────────────────────────────────────────────────────┤
│ ③ 下载文件（二进制流 / 重定向）                                 │
│    云 → 上位机/网页：exe、test_case zip、日志 zip               │
│    Content-Type: application/octet-stream | application/zip   │
│    或大文件 302 → 对象存储签名 URL                              │
└─────────────────────────────────────────────────────────────┘
```

#### 4.3.1 各业务用什么方式


| 业务 | 方向 | 方式 | 说明 |
|------|------|------|------|
| 登录 / Token | 双向 | **JSON** | `POST /auth/login` 等 |
| 阈值拉取 / 发布 | 云→上位机、网页↔云 | **JSON** | 键值在 `data.items[]` |
| OTA **检查** | 云→上位机 | **JSON** | `GET /host-app/check`，含 `downloadUrl`、`sha256` |
| OTA **下载 exe** | 云→上位机 | **二进制 GET** | `GET /host-app/download/{buildId}` 或 check 里的 URL |
| 用例 **manifest** | 云→上位机 | **JSON** | 文件名、version、sha256 |
| 用例 **bundle** | 云→上位机 | **zip 流** | `GET /test-cases/bundle` |
| 用例 **单 ini**（网页保存） | 网页→云 | **JSON** 或 `text/plain` | 正文放 body，元数据 JSON（推荐 `PUT` body 为 ini 文本，`Content-Type: text/plain; charset=utf-8`） |
| **日志上传** | 上位机→云 | **multipart** | 字段 `factoryName`、`deviceId`、`station` + 文件 part `file`（zip） |
| **日志下载** | 云→网页 | **zip 流** 或签名 URL | `GET /logs/{id}/download` |
| 统一发布 check | 云→上位机 | **JSON** | 各层是否有更新 |

#### 4.3.2 上传（multipart/form-data）

**适用**：上位机打包后的 **日志 zip**；网页上传 **exe**（管理端）。

```http
POST /api/factory-tool/logs/upload
Content-Type: multipart/form-data; boundary=----...
Authorization: Bearer <token>    ; 接入登录后
Device-Id: DESKTOP-LINE01

------...
Content-Disposition: form-data; name="deviceId"

DESKTOP-LINE01
------...
Content-Disposition: form-data; name="station"

自由工站
------...
Content-Disposition: form-data; name="factoryName"

hz
------...
Content-Disposition: form-data; name="file"; filename="DESKTOP-LINE01_所有log_20260616.zip"
Content-Type: application/zip

<二进制 zip>
------...--
```

- 文本字段：`factoryName`（**必填**，与上位机 `Mes/FACTORY` 同源）、`deviceId`、`station`、可选 `sn`、`testResult`、`clientVersion`。  
- 文件 part 名固定 **`file`**（与现 `log_upload_service` 一致）。  
- **响应仍为 JSON**：`{ "code": 0, "message": "ok", "data": { "logId": "..." } }`。

**exe 管理端上传**（网页）：`POST /admin/host-app/versions` 可用 **multipart**（`file` + `appVersion`、`buildId`、`releaseNotes` 等字段），或 **先传 OSS、再 JSON 提交元数据**（大文件推荐后者）。

#### 4.3.3 下载（二进制）

**适用**：exe、test_case zip、日志 zip。

```http
GET /api/factory-tool/host-app/download/20260616
Authorization: Bearer <token>
Device-Id: DESKTOP-LINE01
```

- **响应体**：文件原始字节；`Content-Type: application/octet-stream` 或 `application/zip`。  
- **响应头**：`Content-Disposition: attachment; filename="new_production_20260616.exe"`。  
- **大文件**：接口返回 **302**，`Location` 为对象存储 **短时签名 URL**；上位机跟随重定向下载（与现网 curl/Qt 行为一致）。  
- **完整性**：OTA / bundle 在 JSON check/manifest 中带 **sha256**；上位机落盘后校验，**不信任仅 HTTP 200**。

**不用 JSON 包一层 Base64 传 exe/zip**（体积膨胀、占内存）；元数据 JSON、文件本体走流或 OSS。

#### 4.3.4 上位机实现要点（Qt）

| 操作 | 实现 |
|------|------|
| JSON 请求 | `QNetworkAccessManager::post/get`，body `QJsonDocument` |
| multipart 上传 | `QHttpMultiPart` + `QHttpPart`（现 `log_upload_service`） |
| 文件下载 | `QNetworkAccessManager::get` 写 `QFile`；校验 sha256 |
| 公共头 | `factory_cloud_client` 统一加 `Authorization`、`Device-Id` 等 |

#### 4.3.5 网页实现要点

| 操作 | 实现 |
|------|------|
| JSON API | axios，`Content-Type: application/json` |
| 上传 exe | `FormData` + `multipart/form-data` |
| 下载日志 | `window.open` 或 axios `responseType: 'blob'` |

细节见 [后端设计 §5.4](./云端工厂平台-后端设计.md)、[前端设计](./云端工厂平台-前端设计.md)。

### 5.1 日志上传

**接口**：`POST /api/factory-tool/logs/upload`  
**Content-Type**：`multipart/form-data`


| 字段            | 类型        | 必填  | 说明             |
| ------------- | --------- | --- | -------------- |
| factoryName   | string    | 是   | 工厂名；上位机取自 `Mes/FACTORY`（MES 设置页「工厂」下拉） |
| deviceId      | string    | 是   | 上位机 ID，默认主机名   |
| station       | string    | 是   | 工站名/工位         |
| file          | file(zip) | 是   | `所有log` 打包 zip |
| sn            | string    | 否   | 当前测件 SN，便于检索   |
| testResult    | string    | 否   | PASS/FAIL      |
| clientVersion | string    | 否   | exe 版本         |


**上位机取值**：`factoryName` = `SETTINGS.value("Mes/FACTORY")`；未选工厂时 **禁止上传** 并提示先配置 MES 工厂。

**云端处理**：入库 `log_archives.factory_name`；网页列表/筛选展示工厂（代码映射为中文名，见后端 `factories` 字典）。

**网页在线查看**（仍上传 zip）：服务端接收后 **解压索引**，网页按文件树预览 `.txt`；整包下载保留原 zip。

**上位机流程**（与现实现一致）：

1. 测试结束或设置页手动触发
2. 读取 `Mes/FACTORY` → `factoryName`
3. `compressLogDirectory` → zip
4. `uploadLogArchive` → multipart POST（含 `factoryName`）
5. 成功/失败提示；失败可入队重试（本地 `pending_uploads/` 队列，**二期**）

**配置键**：`RemoteLog/UploadUrl`、`RemoteLog/DeviceId`（已有）；`factoryName` 与 `Mes/FACTORY` 同源，不单独建键。

### 5.2 测试数据（能力 #2）

**目标**：产测分项结果 JSON 上云，供网页查询与导出；**不替代** CSV、本地库、各厂 MES。

**做**：

- `POST /api/factory-tool/test-records`（可匿名或带 Token，见后端实现）
- 网页「数据查询 → 测试数据」：按工厂、工站、SN、时间筛选；详情与导出
- 上位机：`TestDataUploadService`；本地仍写 `platform/cloud/test_record/test_record_store` → `所有log/test_pass_data.db`
- **不过站也上报**（如自由工站未勾选 MES 时仍上传云端）

**不做**：

- 上传原始 CSV / zip 测试报表文件
- 代理或替代 `agreement/qmes/*` 过站协议
- 与客户 MES 出货报表合并（属 #8 待立项）

**字段**：对齐 `MesPacketData.itemvalue` / `TestRecordStore::parseItemValue` 分项结构（name、value、max/min、result 等）。

**配置键**：`FactoryCloud/Feature/TestDataUpload`；工厂名同源 `Mes/FACTORY`。

### 5.3 阈值下发

**目标**：网页改 RSSI、电流、电量等卡控；上位机拉取后写入 `SETTINGS`，与 `qsetting::loadConfig` 同源。

**建议接口**：

`GET /api/factory-tool/thresholds?stationKey=FREE_WORK&productModel=M8`

```json
{
  "code": 0,
  "data": {
    "version": "20260612-003",
    "updatedAt": "2026-06-12T10:00:00Z",
    "items": [
      { "settingsKey": "BLE/LowRssi", "value": -60 },
      { "settingsKey": "BLE/HighRssi", "value": -30 },
      { "settingsKey": "Current/LowCharCurrent", "value": 5 },
      { "settingsKey": "Current/HighCharCurrent", "value": 120 }
    ]
  }
}
```

**上位机流程**：

1. 启动或设置页「从云端同步阈值」
2. 比较 `FactoryCloud/ThresholdVersion` 与云端 `version`
3. 有更新则逐项 `SETTINGS.setValue` 并 `saveConfig`
4. 工站构造函数里已缓存阈值的，需 **重新 load 或发信号刷新**（实现时注意与 `test_base` 卡控一致）

**原则**：键名必须与现有 `上位机设置.ini` **完全一致**（`分类/键名`），网页端用下拉选已有键，避免拼写错误。

### 5.4 上位机 OTA 升级

**目标**：替代 `box_base::checkAndUpdateFile()` / `MainWindow::checkAndUpdateFile()`（`mainlogic.cpp`）扫 HTML 目录；对接 `fctp.luteos.com`，支持 **sha256 校验、灰度、强制升级**；与 `AbIni.h` 工站版本号、`test_case` / 阈值云同步分工明确。

**模块**：`platform/cloud/host_ota_service.`*（复用现有 `delete_self.bat` 替换 exe 思路，禁止未校验直接 `startDetached`）。

#### 5.4.1 现状与问题（仓库内已有）


| 概念    | 来源                                           | 示例                            | 现状用途                          |
| ----- | -------------------------------------------- | ----------------------------- | ----------------------------- |
| 工站显示版 | `my_set/AbIni.h` → `#define XXX_VER`         | `自由测试工站 V1.1.14`              | 界面、`upperComputerVer`、CSV、MES |
| 构建包名  | `new_production.pro` → `TARGET`              | `new_production_20260616.exe` | 编译日产出                         |
| 运行工站  | `SETTINGS` → `TestOrderMeta/SelectedStation` | `FLOW_ST_0004`                | 流程、云端 `stationKey`            |


旧 OTA（`http://163.177.79.53:16888/versions/`）问题：

- 明文 HTTP + 正则爬 HTML 目录，无完整性校验；
- 用 **bin 目录内 exe 文件名日期** 比新旧，与 **正在运行的 exe**、`**V1.x.x` 语义版本** 脱节；
- `mainwindow` 与 `box_base` 双份实现；入口受 `SYSTEM/ShowUpperComputerOTAFunc` 控制（产线 ini 多为 `false`）。

本仓库为 **一份 exe 多工站**（`main.cpp` 加载各 `*box`），OTA 按 **整包** 升级，不按工站拆 exe。

#### 5.4.2 版本模型（OTA 统一认这套）

每个可发布包在云端与上位机本地均使用 **双字段**：


| 字段           | 含义                 | 示例                          | 维护方                            |
| ------------ | ------------------ | --------------------------- | ------------------------------ |
| `appVersion` | 语义版本，给人看、给日志/MES   | `1.1.15`                    | 发版时 bump，可与 `DEBUG_VER` 或主工站对齐 |
| `buildId`    | 构建标识，OTA 判定「有没有新包」 | `20260616` 或 `20260616.001` | 编译日或 CI 流水号，**单调递增**           |


**比对规则**：先比 `buildId`；相同再比 `appVersion`（semver）。同一天打多包时 `buildId` 用 `yyyyMMdd.N`。

**本地持久化**（`上位机设置.ini`，check 成功或升级完成后写入）：

```ini
HostAppVersion=1.1.14
HostAppBuildId=20260611
HostPackageName=new_production
```

**请求头**（与其它 Factory API 一致，见 §4.1）：

```http
APP-Version: 1.1.14
X-Client-Build: 20260611
Station-Key: FLOW_ST_0004
```

check 请求建议附带：`packageName`、`buildId`、`appVersion`、`stationKey`、`deviceId`（query 或 JSON）。

#### 5.4.3 `XXX_VER` 与 OTA 的分工


| 层级     | 字段                                   | 用途                            | OTA 是否参与比对                  |
| ------ | ------------------------------------ | ----------------------------- | --------------------------- |
| 整包 OTA | `appVersion` + `buildId`             | 下载、替换 exe                     | **是**                       |
| 工站展示   | `FREE_VER`、`PCBA_VER` 等              | 界面、CSV、MES、`upperComputerVer` | **否**（同 exe 内多工站展示版本，不宜单独比） |
| 测试用例   | `FactoryCloud/TestCaseBundleVersion` | `test_case` 云同步               | 否，走 §5.5                    |
| 卡控阈值   | `FactoryCloud/ThresholdVersion`      | SETTINGS 云同步                  | 否，走 §5.3                    |


**发版节奏建议**：

- 改某工站逻辑 → 对应 `XXX_VER` patch +1，整包 `appVersion` patch +1，`buildId` = 当天；
- 只改 `test_case` / 阈值 → **不发 exe**，只升 bundle / threshold version；
- 公共模块大改（`test_base`、`agreement` 等）→ 提交说明对各 `XXX_VER` 各写一行（见 `docs/上位机版本信息生成说明.md`），整包 `appVersion` 至少 patch +1。

#### 5.4.5 上位机流程

```text
启动后 / 菜单「软件更新」
  → host_ota_service::checkUpdate()
  → GET /host-app/check（带本地 buildId、appVersion、stationKey）
  → 有新版本：弹窗 releaseNotes（forceUpgrade 则测试空闲时强制）
  → 下载到 bin/updates/
  → 校验 sha256
  → 启新 exe + helper.bat 删除当前 exe（沿用 delete_self.bat 模式）
  → 更新 SETTINGS：HostAppVersion、HostAppBuildId
```


| 触发时机                      | 行为                      |
| ------------------------- | ----------------------- |
| 启动后延迟（可配置，如 30s）          | 静默 check，有更新再提示         |
| 菜单「软件更新」、`on_downloadapp` | 手动 check（保留现有入口，内部切新服务） |
| `forceUpgrade=true`       | 未在测试中时可阻断并提示必须升级        |


**安全**：HTTPS + sha256；禁用明文 HTTP 与 HTML 目录爬取；`163.177.79.53` 方案仅作过渡期 fallback，稳定后关闭。

#### 5.4.6 灰度与发布通道（云端配置）

| `deviceId` 白名单 | 内测机台        |
| `forceUpgrade` | 协议/安全不兼容时强制 |


网页「上位机版本」页：上传 exe、填写 `appVersion` / `buildId` / `releaseNotes` / sha256，配置灰度规则（见 §5.7）。

#### 5.4.7 与其它「升级」能力的边界


| 能力          | 模块                       | 版本对象                      |
| ----------- | ------------------------ | ------------------------- |
| **上位机 exe** | `host_ota_service`       | `new_production_*.exe`    |
| **设备固件**    | `root_ble_ota`、主界面 OTA 页 | 手柄 / dongle 固件（**不在本方案**） |

四者分开规划；升 exe **不等于** 用例或阈值已更新。

#### 5.4.8 研发发版操作清单

### 5.6 登录与账号管理

**定位**：为日志上传、阈值/用例同步、上位机 OTA 等 **`/api/factory-tool/*`** 提供鉴权。

#### 5.6.1 接口


| 方法   | 路径                               | 说明                           |
| ---- | -------------------------------- | ---------------------------- |
| POST | `/api/factory-tool/auth/login`   | 用户名密码 → `accessToken` + 过期时间 |
| POST | `/api/factory-tool/auth/refresh` | 刷新 Token                     |
| GET  | `/api/factory-tool/auth/me`      | 当前用户、角色、可访问工站列表              |
| CRUD | `/api/factory-tool/admin/users`  | **仅网页管理端**                   |


**login 请求体**（上位机登录时**必须上报当前电脑名**，供后台区分产线 PC、绑定会话与审计）：


| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| username | string | 是 | 工厂云账号 |
| password | string | 是 | 密码 |
| hostName | string | 是 | **当前电脑名**（Windows「计算机名」），默认 `QSysInfo::machineHostName()` |
| deviceId | string | 否 | 设备标识；**缺省时与 `hostName` 相同**；可与 `FactoryCloud/DeviceId` 配置一致 |
| stationKey | string | 否 | 当前工站键，对齐 `TestOrderMeta/SelectedStation` |

**login 请求示例**：

```json
{
  "username": "line01",
  "password": "***",
  "hostName": "DESKTOP-LINE01",
  "deviceId": "DESKTOP-LINE01",
  "stationKey": "FLOW_ST_0004"
}
```

> 实现上与日志上传 `deviceId` 同源：`LogUploadService::defaultDeviceId()` → `QSysInfo::machineHostName()`；若 `FactoryCloud/DeviceId` 或 `RemoteLog/DeviceId` 已配置非空，登录时 `deviceId` 用配置值，**`hostName` 仍上报真实电脑名**。

**login 响应示例**：

```json
{
  "code": 0,
  "data": {
    "accessToken": "...",
    "expireAt": "2026-06-17T08:00:00+08:00",
    "roles": ["operator"],
    "stationKeys": ["FLOW_ST_0004"]
  }
}
```

#### 5.6.2 角色与权限


| 角色 | 上位机（产测 Qt） | 网页管理端 |
|------|-------------------|------------|
| **operator** | 登录；上传日志；手动「检查更新」（OTA 提示） | 只读：本工站日志查询 |
| **engineer** | operator 全部；从云端同步阈值、测试用例 | 编辑阈值模板、测试用例、发布 bundle；不可管账号 |
| **admin** | engineer 全部（可选是否允许产线机登录 admin） | 账号 CRUD、工站授权、发版 OTA、强制升级 |

**能力级权限对照**（后端鉴权 + 上位机 UI 按角色隐藏按钮）：


| 能力 | operator | engineer | admin |
|------|:--------:|:--------:|:-----:|
| `auth/login` | ✓ | ✓ | ✓ |
| 日志上传 | ✓ | ✓ | ✓ |
| 阈值同步 | — | ✓ | ✓ |
| 用例同步 | — | ✓ | ✓ |
| OTA 检查/下载 | ✓（可选灰度） | ✓ | ✓ |
| `admin/users` CRUD | — | — | ✓ |

默认产线工位使用 **operator**；工艺工程师使用 **engineer**；**admin** 仅限办公室账号，**不建议**在产线 PC 长期登录。

#### 5.6.3 账号管理策略（总则）

1. **集中管理**：账号仅在 **网页管理端** 由 `admin` 创建/停用/改密/授权；**无上位机自助注册、无找回密码入口**（走内部 IT/工艺管理员线下处理）。  
2. **最小权限**：账号绑定 **角色** + **可访问工站列表**（`stationKeys`）；登录时若请求的 `stationKey` 不在授权列表内 → `403`。  
3. **人机绑定**：每次登录上报 **`hostName`（电脑名，必填）** 与 **`deviceId`**；后台记录「谁、在哪台 PC、哪个工站」登录，供审计与排障。  
4. **产测优先**：云端登录失败或 Token 过期 **不得阻断** 本地产测（`FactoryCloud/RequireOnline=false` 为默认）；仅云上传/同步/OTA 等能力不可用。  
5. **一机多账号 / 一账号多机**：  
   - **允许多台 PC** 使用同一 operator 账号（如一条产线共用一个工艺账号），后台按 `hostName` 区分会话；  
   - **同一 PC** 换班可换账号登录，旧 Token 失效或允许多会话（由后端策略定，建议 **单 PC 仅保留最新会话**）。  
6. **与 MES 工号无关**：工厂云账号 **不等同** MES `userNo` / 员工工号；MES 仍走各厂 `qmes` 配置，两者仅在报表中可关联展示。

#### 5.6.4 账号生命周期与密码策略


| 阶段 | 策略 |
|------|------|
| **创建** | `admin` 在网页填写：用户名、初始密码、角色、`stationKeys`、备注（产线/工厂）；初始密码 **首次登录强制修改**（网页或上位机弹窗，二期） |
| **启用/停用** | 停用后该账号 **立即无法 login**；已签发 Token 在下次校验时失效 |
| **改密** | 用户改密（知道旧密码）或 `admin` 重置；重置后可选强制下线所有会话 |
| **删除** | 建议 **逻辑删除**（保留审计）；物理删除仅 `admin` 且无关联审计要求时 |
| **密码强度** | 最少 8 位，含字母与数字（后端校验）；定期更换 **建议 90 天**（可配置，二期提醒） |
| **锁定** | 连续失败 **5 次** / 15 分钟 → 锁定 30 分钟或 `admin` 手工解锁 |

**用户名约定**（建议）：`{工厂缩写}_{产线}_{序号}`，如 `hz_line01_op`。

#### 5.6.5 工站授权与产线 PC


| 规则 | 说明 |
|------|------|
| **工站授权** | 账号的 `stationKeys` 为白名单；上位机 `SETTINGS` 里当前 `TestOrderMeta/SelectedStation` 须在名单内方可登录成功 |
| **登录时校验** | `login` 请求带 `stationKey`；后端校验与 `stationKeys` 交集；响应可返回 **允许工站全集** 供设置页展示 |
| **电脑名** | `hostName` **始终为 OS 计算机名**，不可由 ini 伪造；用于后台「设备列表」与异常登录发现 |
| **deviceId** | 默认等于 `hostName`；特殊机台可配置 `FactoryCloud/DeviceId` 作为稳定别名，但 `hostName` 仍上报 |
| **设备登记**（可选） | 网页维护「已登记 PC」：`hostName` + 产线 + 工位；未登记 PC 登录时 **告警但不阻断**（或 admin 配置为阻断，二期） |

#### 5.6.6 Token 与会话策略


| 项 | 建议值 | 说明 |
|----|--------|------|
| `accessToken` 有效期 | **8～12 小时** | 覆盖一个班次；过期需重新 login 或 refresh |
| `refreshToken` 有效期 | **7～30 天** | 仅上位机本地保存；用于无感续期（二期） |
| 传输 | **HTTPS only** | 与 §4.1 一致 |
| 存储（上位机） | `FactoryCloud/Token`、`TokenExpireAt` | 可选 DPAPI/简单混淆存盘；**默认不落盘明文密码** |
| 登出 | 设置页「退出登录」 | 清空本地 Token；可选调 `POST /auth/logout` 作废服务端会话 |
| 401 处理 | 任意 factory-tool 接口返回 401 | 清空 Token → 提示重新登录；**不退出产测流程** |

**refresh 请求示例**（二期）：

```json
{
  "refreshToken": "...",
  "hostName": "DESKTOP-LINE01",
  "deviceId": "DESKTOP-LINE01"
}
```

#### 5.6.7 上位机登录行为与配置

**设置页「云端工厂」登录区**：

| UI 项 | 行为 |
|-------|------|
| 用户名 / 密码 | 对应 `FactoryCloud/AuthUser`、`AuthPassword`；保存配置时 **可选仅记住用户名** |
| 当前电脑名 | 只读，来自 `QSysInfo::machineHostName()` |
| 登录状态 | 显示已登录用户、角色、`TokenExpireAt`、上次登录时间 |
| 登录 / 退出 | 调 `auth_service::login` / `logout` |

**配置项策略**：


| 键 | 默认 | 说明 |
|----|------|------|
| `FactoryCloud/RequireOnline` | `false` | `true`：未登录则禁用云上传/同步（产测仍可跑） |
| `FactoryCloud/RememberUsername` | `true` | 是否保存用户名 |
| `FactoryCloud/RememberPassword` | `false` | **默认 false**；若 true 则密码写入 ini（仅内网受控机台，需现场规范） |
| `FactoryCloud/DeviceId` | 空 | 空则用电脑名 |

**实现要点**：

- 提交 login 前读取 **`QSysInfo::machineHostName()`** → `hostName`（必填）；`deviceId` 优先 `FactoryCloud/DeviceId`，空则同 `hostName`。  
- 登录成功：写 `Token`、`TokenExpireAt`、`AuthUser`；`showlog` 记录「工厂云登录成功：用户 xxx，电脑 xxx」。  
- 后续 `platform/cloud/*`：请求头 **`Device-Id`** + **`Authorization: Bearer <Token>`**。  
- 日志上传等已上线接口：过渡期可匿名；接入登录后与 `deviceId` / `hostName` 一致。

#### 5.6.8 审计与安全

**后台记录**（建议字段）：`timestamp`、`username`、`hostName`、`deviceId`、`stationKey`、`action`（login/logout/login_fail/token_refresh）、`ip`、`clientVersion`。

**网页「账号管理」页能力**：

- 用户列表：角色、工站、状态、最后登录时间、最后登录 `hostName`  
- 新建 / 停用 / 重置密码 / 编辑 `stationKeys`  
- 登录失败次数与锁定状态  
- **不在网页展示** 产线 operator 的明文密码（仅创建/重置时显示一次）

**上位机安全**：

- 密码禁止写进 `showlog` 或上位机 log 文件  
- Token 过期、403 权限不足须 **明示原因**（工站未授权 / 账号停用等）  
- 产线 PC 磁盘若启用 `RememberPassword`，需在交付规范中注明风险
