# 上位机版本发布说明如何生成

## 原理简述

1. **`my_set/AbIni.h`** 中每一条 `#define XXX_VER "显示名称   Vx.y.z"` 会对应发布说明里的**一行**。脚本只认这种宏名；提交信息里写的 `[XXX_VER]` 必须与这里的宏名**完全一致**才会被归类。
2. **`docs/生成版本说明.py`** 在仓库根目录执行 `git log`，时间范围由脚本内 **`SINCE_DATE`**、**`UNTIL_DATE`** 决定（默认自 `2026-4-14` 起）。
3. `git log` 的每条提交会取**标题与正文全文**解析；正文中出现的 `[XXX_VER]` 与标题行等效。带 `[XXX_VER]` 的片段会被收集到对应工站显示名称那一行的括号内；**整段提交里若没有任何合法 `[XXX_VER]`**，则该提交**不会**写入各工站行（对应时段内各工站仍可能显示 `(-----)`），这通常用于「仅文档 / 仅生成脚本」类维护提交。该时间段内没有匹配提交时，对应项为 `(-----)`。
4. 生成结果以**追加**方式写入 **`docs/上位机版本发布.md`**（每次运行多一节「上位机版本」块）。若需对外只保留最新一节，可手工删除旧块或自行整理。

## 如何执行

在已安装 Git、且当前目录为仓库根目录的环境下执行：

```text
python docs/生成版本说明.py
```

（若系统使用 `py` 启动器，可为 `py docs/生成版本说明.py`。）

脚本会自动解析仓库根目录；也可在脚本中修改 `REPO_PATH` 指向其他克隆路径。

与提交信息格式约定一致的可参考项目根目录 **`.copilot-commit-message-instructions.md`**。若变更为**公共模块**（各工站共享代码），提交说明应对 **`AbIni.h` 中每一个 `#define XXX_VER` 各写一行**（可放在**提交正文**，每行一个 `[XXX_VER] …`），每行附上**同一套**公共改动说明，以便本脚本把该说明写入发布文档的**每一条**工站记录；**不要**自造 `AbIni.h` 里不存在的 `[XXX_VER]`。

## 提交信息校验（稳定落地，可选）

仅靠 Copilot 提示词容易漏规则，可在本机启用 **commit-msg 钩子**，在 `git commit` 时用暂存路径与提交正文做自动检查（与上文约定、`docs/生成版本说明.py` 对齐）：

1. **每个克隆 / 本机工作副本只需配置一次**（`git pull` 后一般不必重做）：在仓库根目录执行 `git config core.hooksPath .githooks`，或运行 **`scripts/setup-git-hooks.cmd`**（Windows 命令提示符）/ **`scripts/setup-git-hooks.ps1`**（需用 PowerShell 调用）/ **`scripts/setup-git-hooks.sh`**（Git Bash 等）。说明：`core.hooksPath` 写在**本机** `.git/config`，**不能**只靠拉代码自动生效；仓库内的 **`.githooks` 目录会随 clone 下来**，但需要上述一步把 Git 指过去。**若未配置，Git 不会执行本仓库的 `commit-msg`，界面点提交也不会被拦截。** 可用 `git config --get core.hooksPath` 检查，应输出 `.githooks`。
2. 钩子会调用 **`docs/validate_commit_message.py`**：检查 `[XXX_VER]` 是否在 `AbIni.h` 中有定义、是否禁止 `AGREEMENT_VER`、同一行是否出现多个标签、以及**按路径是否漏写**应有的标签（如 `work_station/ageing/` → `[AGE_VER]`，`agreement/qtuple/` → `[FREE_VER]` 等）。
3. **跳过**：紧急可用 `git commit --no-verify`，或设置环境变量 `SKIP_COMMIT_MSG_VALIDATE=1`。
